**A-03｜快照的寫入時機與重新整理規則**

| 欄位 | 內容 |
|---|---|
| **待確認事項** | 快照在何時寫入？出貨單建立後、尚未完成（`done`）前，是否允許重新整理快照？ |

| **顧問建議** | 採選項 1 為 MVP 預設。選項 2 可列為第二階段功能，避免意外覆蓋已確認的快照內容。選項 3 風險高，不建議 |
| **影響範圍** | `create()` 邏輯設計、是否需要「重新整理」按鈕開發、快照的可信度與追溯性 |
| **優先度** | 🔴 最高 |

**A-04｜資料儲存方式：快照 vs 即時讀取**

| 欄位 | 內容 |
|---|---|
| **待確認事項** | 出貨單上的交貨條件是複製一份文字快照，還是即時讀取主檔？ |
| **建議選項** | 選項 1：快照（建單時複製，之後與主檔脫鉤）（顧問建議）；選項 2：即時讀取（主檔修改後，所有已建出貨單的顯示也會改變） |
| **顧問建議** | 採選項 1。出貨行為發生後，條件文字應被固定，以利事後追溯與稽核。選項 2 存在「出貨後才發現條件被改動」的風險 |
| **影響範圍** | Model 欄位設計、資料一致性、稽核可追溯性 |
| **優先度** | 🔴 最高 |

**B-01｜`date_start` 必填性與預設值**

| 欄位 | 內容 |
|---|---|
| **待確認事項** | `date_start` 是否設為必填？預設值是否帶入今天？ |
| **建議選項** | 選項 1：必填，預設帶入今天（顧問建議）；選項 2：非必填，空白 = 立即生效；選項 3：非必填，空白 = 不納入查詢 |
| **顧問建議** | 優先採選項 1。必填＋預設今天兼具嚴謹性與便利性，既避免空白歧義，又不增加維護阻力。若客戶不接受必填，才進一步討論選項 2 或 3 |
| **影響範圍** | 欄位驗證邏輯、有效條件查詢的 WHERE 條件 |
| **優先度** | 🟠 高 |

**B-02｜有效日期重疊的系統處理**

| 欄位 | 內容 |
|---|---|
| **待確認事項** | 同一客戶若建立兩筆有效日期重疊的條件，系統應阻擋、發出警告，還是允許並由使用者自行判斷？ |
| **建議選項** | 選項 1：阻擋儲存，強制日期不重疊；選項 2：允許但發出警告提示；選項 3：完全允許，不做驗證（顧問建議若 A-02 確認允許多筆，採此選項） |
| **顧問建議** | 若 A-02 採「允許多筆合併顯示」，建議不阻擋重疊（選項 3）；若 A-02 採「只取最新一筆」，建議至少發出警告（選項 2） |
| **影響範圍** | `_sql_constraints` 或 `@api.constrains` 設計 |
| **優先度** | 🟠 高（依 A-02 結果決定） |

**B-06｜`date_end` 空白 = 永久有效的確認**
