## 一、【需求說明】

1. 部分客戶對出貨方式、包裝方式、交貨時間有特殊要求，目前以口頭或 Email 傳達，無法在系統中集中查閱，容易造成出貨錯誤或遺漏。
2. 新增「客戶特殊交貨條件」作業，集中記錄每位客戶的特殊交貨需求，供業務與倉管人員出貨前確認。

---

### 需求前置分析：三種設計方向評估

> 本作業在正式規格確認前，需先與客戶確認以下三種設計方向，再決定採用何種方式。選擇不同方向將影響後續開發範圍與複雜度。

**方向 A：客戶主檔頁籤**
在 `res.partner`（客戶主檔）新增一個「特殊交貨條件」頁籤，於客戶頁面直接維護。
- 優點：查找直覺、與客戶資料整合度高
- 限制：無法記錄多筆或多時段不同條件；適合「一客戶一組條件」的簡單場景
- 開發範圍：繼承 `res.partner`，新增頁籤與欄位

**方向 B：獨立特殊交貨條件主檔**（本草稿預設方向）
建立獨立的 Odoo Model，每筆記錄對應一個客戶的一組交貨條件，可設定有效日期範圍，支援同一客戶多筆不同時期的條件。
- 優點：彈性高、可追蹤歷史、可設有效期
- 限制：需獨立維護，若不與訂單 / 出貨單串接，仍依賴人工查閱
- 開發範圍：新增 Model + View，若需串接出貨單則須額外開發

**方向 C：報價單 / 出貨單自動帶出提示**
在報價單或出貨單選取客戶時，自動查詢並顯示該客戶的有效特殊交貨條件。
- 優點：在操作流程中自動提醒，降低人工遺漏風險
- 限制：需額外開發觸發邏輯，通常依賴方向 A 或 B 作為資料來源
- 開發範圍：方向 A 或 B 為前提，加上 `sale.order` 或 `stock.picking` 的繼承開發

> **建議：本草稿以方向 B 為主進行規格設計，但需於下次確認會議決定是否同時開發方向 C 的串接提示功能。**

### 顧問建議 MVP 方案

**第一階段建議：方向 B（獨立特殊交貨條件主檔）**

建議第一階段採方向 B，建立獨立主檔作業。若第一階段只完成「可新增 / 查詢主檔」，而完全不串接銷售訂單或出貨單，則使用者仍需自行至主檔查閱，**無法有效降低出貨遺漏風險**，可能與原始需求的核心目標相悖。

**MVP 最小建議串接範圍：**

| 階段 | 串接目標 | 說明 |
|---|---|---|
| **第一階段（MVP）** | 出貨單（`stock.picking`）帶出提示 | 倉管操作出貨時，自動顯示該客戶的有效特殊交貨條件 |
| 第二階段 | 報價單（`sale.order`）帶出提示 | 業務開單時可查看客戶條件 |
| 第二階段 | 銷售訂單確認提示 | 確認訂單時顯示警示 |
| 第二階段 | 列印出貨單帶出備註 | 紙本出貨單上印出特殊交貨條件文字 |

---

### MVP 範圍明確定義

> **MVP 定義：**
> 建立獨立「客戶特殊交貨條件」主檔作業（方向 B），並於出貨單（`stock.picking`）建立時，依對應客戶自動查詢目前有效的特殊交貨條件，將提示文字以快照方式寫入出貨單指定欄位，供倉管人員出貨前確認。

MVP 包含的交付項目：

| 項目 | 說明 |
|---|---|
| 客戶特殊交貨條件主檔作業 | 含 TreeView + FormView，支援新增 / 修改 / 封存 |
| 出貨單客戶有效條件查詢邏輯 | 出貨單建立時，自動查詢主檔中目前有效的條件 |
| 出貨單快照欄位 | `customer_delivery_condition_snapshot`，自動寫入有效條件文字 |
| 出貨現場補充備註欄位 | `customer_delivery_condition_note`，允許倉管手動補充 |

### 本階段不處理範圍

| 本階段不處理的功能 | 說明 |
|---|---|
| 報價單提示 | 報價單選定客戶時不自動帶出特殊交貨條件 |
| 銷售訂單確認警示 | 銷售訂單確認時不觸發警示或彈窗 |
| 出貨單列印帶出 | 紙本或 PDF 出貨單上不印出特殊交貨條件文字 |
| 主管簽核流程 | 條件新增 / 修改不需主管審核，直接生效 |
| 倉管異動申請通知 | 倉管無法在系統內發起條件異動申請 |
| 封存原因記錄 | 封存僅使用 Odoo 標準 `active` 欄位行為，不記錄封存原因 |
| 封存簽核 | 封存不需主管批准，業務主管可直接執行 |

| **主管審核機制** | 是否需要主管審核才能讓條件生效？若需要，屬自訂開發範疇 | 是否引入 workflow / 狀態機 |

### 單據帶出規則確認

| 項目 | 待確認內容 | 兩種可能設計 |
|---|---|---|
| **自動查詢時機** | 出貨單在建立時自動查詢有效條件並寫入快照 | `create()` 觸發 |
| **多筆有效條件顯示方式** | 若同一客戶同時有多筆有效條件，系統顯示全部合併，還是只取最新一筆？ | 合併全部 / 取最新 / 彈窗選擇 |
| **條件資料的儲存方式** | 帶出的條件是複製一份文字到出貨單上（快照），還是即時讀取主檔？ | 快照（顧問建議）vs. 即時讀取 |
| **主檔修改後的影響** | 主檔條件被修改後，已建但尚未出貨的出貨單是否需要同步更新提示？ | 自動同步 vs. 快照保留 |
| **歷史條件保留需求** | 出貨完成後，是否需要在出貨記錄中保留當時帶出的條件文字作為稽核依據？ | 若需要，單據上需有獨立快照欄位 |

> **顧問建議：** 採「快照」設計，出貨行為發生後條件文字應被固定，不因主檔後續修改而變動，以利日後問題追溯。

## 附錄：出貨單串接欄位設計（v0.5）

> 本附錄為 MVP 串接出貨單所需的新增欄位設計。繼承 `stock.picking`（出貨單）。

### 欄位一：客戶特殊交貨條件快照（系統自動寫入）

| 欄位項目 | 內容 |
|---|---|
| **中文欄位名稱** | 客戶特殊交貨條件快照 |
| **技術欄位名稱** | `customer_delivery_condition_snapshot` |
| **型態** | C2（Text，多行文字） |
| **唯讀** | 是（系統自動寫入，倉管不得修改，確保快照內容可信） |
| **必輸** | 否（客戶若無有效條件則欄位為空或填入提示文字） |
| **追蹤** | 是（需工程師確認 `stock.picking` 是否支援欄位層級 tracking；若不支援，改以 log note 記錄） |
| **資料處理** | 出貨單建立（`create()`）時，依對應客戶自動查詢有效特殊交貨條件並寫入；出貨完成後不因主檔後續修改而變動，用於出貨現場提示與事後追溯 |

### 欄位二：出貨現場補充備註（倉管手動填寫）

| 欄位項目 | 內容 |
|---|---|
| **中文欄位名稱** | 出貨現場補充備註 |
| **技術欄位名稱** | `customer_delivery_condition_note` |
| **型態** | C2（Text，多行文字） |
| **唯讀** | 否（允許倉管依現場狀況手動填寫） |
| **必輸** | 否 |
| **追蹤** | 是（記錄填寫者與時間） |
| **資料處理** | 無預設邏輯，倉管自行填寫；不影響快照欄位內容 |

### 快照觸發邏輯（開發參考）

```
觸發時機：create()（出貨單建立時）
查詢邏輯：
  1. 依 partner_id 查詢自訂 Model
  2. 過濾條件：
     active = True
     AND date_start <= today
     AND (date_end >= today OR date_end IS NULL)
  3. 若有多筆有效條件：
     依 date_start 由新到舊排序，全部合併顯示
     （各筆條件間以換行分隔，並標示各自起始日）
  4. 將合併結果文字寫入 customer_delivery_condition_snapshot
  5. 若查無有效條件：填入「無特殊交貨條件（系統查無有效設定）」（待確認）
```

### FormView 顯示建議

- 於出貨單 FormView 的「附加資訊」頁籤或「備註」區塊新增兩個欄位
- 快照欄位標籤建議標示「⚠️ 客戶特殊交貨條件（出貨前請確認）」
- 兩欄視覺上應有所區分，快照欄位建議底色或字色與一般備註欄位有所不同（待確認視覺設計規範）
