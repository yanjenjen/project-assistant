## 3. 研發流程節點

| 流程節點 | 作業摘要 | 後續 ERP 顧問觀察 |
|---|---|---|
| 提案評估 | 營業或其他單位提出新產品提案，研發審查產品功能特性與客戶需求，召開可行性評估會議 | 是否有提案入口、需求來源、可行性評估與跨單位審查紀錄 |
| 設計規劃 | 依設計開發管制表與設計開發流程圖規劃新產品，分派設計任務、建立企劃概要與開發日程 | 是否有設計任務、開發計劃、日程與責任分派資料 |
| 試作樣品 | 依新產品提案與需求表試作手工樣品、工程樣品，進行檢測與試驗評估 | 是否有樣品試作、檢測方法、允收標準與試作討論紀錄 |
| 設計驗證 | 對重要階段功能特性進行測試與驗證，由研發測試、品管驗證 | 是否有驗證紀錄、可靠度試驗、測試失效與改善對策 |
| 設計審查 | 依設計開發管制表分初步、期中、最終階段審查，記錄審查結果與矯正措施 | 是否有階段審查、參與成員、設計輸入輸出與進度審查 |
| 量試與驗收 | 新產品需經量試及量試檢討後，才能進行設計驗收與量產 | 是否有量試、量試檢討、驗收與移交流程 |
| 設計變更 | 產品需變更時由提出單位填具申請，經研發與相關單位確認後辦理 | 是否有設計變更申請、風險成本評估、變更驗證與緊急變更 |
| 文件發行保管 | 研發相關文件需經審查、核准、編號、分發與簽收，無效文件撤銷 | 是否有技術文件版本、簽收、保管、失效撤銷與機密管制 |
| 研發管制 | 研發活動須保留適切紀錄，便於追溯與改善 | 是否需追蹤提案至量產的完整研發生命週期 |

## 5. 角色 / 權責 / 簽核點

| 角色 / 單位 | 權責摘要 | 待確認事項 |
|---|---|---|
| 研發單位 | 文件權責單位；負責提案審查、可行性評估、設計規劃、測試、變更與研發管制 | 研發單位內部角色與核准層級需確認 |
| 營業單位 | 提供新產品提案、市場調查、分析及銷售規劃資料 | 與銷售流程需求來源銜接需確認 |
| 品管單位 | 參與可行性評估、產品驗證、測試失效對策與設計驗收 | 品管驗證與品質文件關係需確認 |
| 生技單位 | 接收最終規格、圖樣、技術資料或樣品，參與設計驗收 | 生技是否為生產技術或工程單位需確認 |
| 採購單位 | 依最終技術資料作業 | 與採購規格資料銜接需確認 |
| 相關單位代表 | 參與設計審查，代表影響品質的相關單位 | 參與單位名單與審查責任需確認 |
| 總經理 / 承認單位 | 文件核准、頒布、修訂或廢止 | 實際核准權限需確認 |

## 6. 主要控制點

### 提案評估

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 新產品提案需由研發審查產品功能特性及客戶需求，作為設計開發基礎 | 權責控制 | 待系統確認 |
| 可行性評估報告應由研發召開審查會議並徵詢營業、品管等相關單位意見 | 單據控制 | 待系統確認 |
| 新產品設計應實施市場調查與可行性評估 | 系統控制 | 待系統確認 |
| 審查未通過者應重新評估檢討 | 例外控制 | 待系統確認 |

### 設計規劃

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 設計規劃活動應依設計開發管制表與設計開發流程圖進行 | 單據控制 | 待系統確認 |
| 研發設計人員與資源需經適當指派及遴選合格人員擔任 | 權責控制 | 待系統確認 |
| 設計規劃與進度管制應確實進行 | 系統控制 | 待系統確認 |
| 設計人員或資源變動時，應評估是否影響設計規劃活動 | 例外控制 | 待系統確認 |

### 試作樣品

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 試作樣品應依新產品提案單及新產品開發需求表進行 | 單據控制 | 待系統確認 |
| 試作樣品應評估檢測、試驗方法、使用狀態與允收標準 | 系統控制 | 待系統確認 |
| 必要時研發單位應針對試作樣品召開內部討論會議 | 權責控制 | 待系統確認 |
| 試作樣品不符規格或技術資料不齊全時，應修訂更正 | 例外控制 | 待系統確認 |

### 設計驗證與審查

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 設計過程重要階段功能特性應加以測試與驗證 | 系統控制 | 待系統確認 |
| 新產品功能測試由研發單位進行，產品驗證由品管單位實施 | 權責控制 | 待系統確認 |
| 測試失效或不符規格時，品管應填製對策報告交研發改善 | 單據控制 | 待系統確認 |
| 設計審查應依設計開發管制表分初步、期中、最終階段實施並作成紀錄 | 單據控制 | 待系統確認 |
| 設計審查應鑑別並預期問題，採取矯正措施 | 例外控制 | 待系統確認 |

### 驗收與變更

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 新產品開發需經量試及量試檢討後，始能進行設計驗收及量產 | 系統控制 | 待系統確認 |
| 設計驗收應由品管及生技單位進行 | 權責控制 | 待系統確認 |
| 最終規格、圖樣、技術資料或樣品應以技術資料移交明細表移交生技單位 | 單據控制 | 待系統確認 |
| 設計變更需填具設計變更申請單，經研發及相關單位確認 | 單據控制 | 待系統確認 |
| 設計變更應考量風險、成本、相關作業配合性與緊急變更處理 | 例外控制 | 待系統確認 |

### 文件發行與保管

| 控制點 | 控制類型 | 系統確認狀態 |
|---|---|---|
| 所有影響產品品質研發活動均應保有適切紀錄，便於追溯及改善 | 單據控制 | 待系統確認 |
| 文件制定、管理、權責劃分及作業應依文件管制規定執行 | 權責控制 | 待系統確認 |
| 文件經審查、核准與編號後始行分發，收發均應有簽收紀錄 | 單據控制 | 待系統確認 |
| 過時無效文件應即刻自所有發行與使用單位撤銷，留用者須標示 | 例外控制 | 待系統確認 |
| 機密文件應被標示並特別管制 | 系統控制 | 待系統確認 |

## 7. ERP 顧問訪談問題

### 提案與評估

1. 新產品提案由哪些單位提出？是否有統一入口與提案格式？
2. 市場調查、銷售規劃與客戶需求如何提供給研發？
3. 可行性評估會議如何召開、參與單位有哪些、決議如何保存？
4. 提案未通過時是否有退回、重評估或結案紀錄？

### 設計規劃與試作

1. 是否有設計開發管制表與設計開發流程圖？是否會隨產品類型不同調整？
2. 新產品開發企劃概要、開發日程、工作分派如何建立與追蹤？
3. 試作樣品、工程樣品、檢測方法與允收標準如何記錄？
4. 技術資料修訂後如何通知相關單位並確認使用最新版？

### 驗證審查與變更

1. 新產品功能測試與品管驗證的分工為何？
2. 新產品驗證紀錄、可靠度試驗、對策報告如何保存與追蹤？
3. 設計審查分哪些階段？每階段輸入、輸出、參與單位與審查紀錄如何控管？
4. 設計變更申請如何發起、核准、驗證與發行？
5. 緊急變更如何避免不合格品生產？

### 文件與跨流程

1. 工程技術資料如何編號、核准、分發、簽收與保管？
2. 過時文件如何撤銷或標示？機密文件如何控管？
3. 最終規格與圖樣如何移交生技、採購、品管或生產相關單位？
4. 研發循環與生產、採購、銷售流程如何銜接，但目前是否仍先保留待系統確認？

## 9. 後續批次萃取格式建議

研發循環可沿用既有內控制度最小萃取格式，10 節結構、第 4 節 6 欄單據格式與第 6 節四類控制點分類均可使用。

本次觀察到的格式議題如下：

| 議題 | 目前處理方式 | 是否需立即升版 |
|---|---|---|
| 研發資料包含提案、需求、設計、驗證、變更與文件管制 | 仍以第 4 節「單據 / 表單 / 紀錄」方式容納 | 否 |
| 研發循環可能橫跨生產、採購、品管與銷售 | 只保留訪談問題與待確認，不做跨循環整合 | 否 |
| 系統可能涉及 PLM / Project / Documents / Quality | 一律標示 `待系統確認`，不推測對應 | 否 |
| 部分紀錄名稱不完整 | 以 `[待確認]` 標示，不補造 | 否 |

建議先完成本草稿格式自檢與使用者驗收，再決定是否更新制度範本總控。若後續投資循環、融資循環等低頻循環出現不同格式需求，再評估格式規則草稿是否升版。
