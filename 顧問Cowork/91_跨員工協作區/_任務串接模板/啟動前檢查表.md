## 使用時機

本檢查表應在以下任一情況發生前完成：

- 建立任何新任務包前
- 接收新客戶資料前
- 將任務從一般討論轉為正式任務包前
- 使用者準備提供真實企業資料前

## 五、敏感資料三級判斷

**A 級 — 禁止寫入任務包任何位置**

以下資料不得以任何形式寫入工作區檔案：

- 客戶帳號、密碼、IP 位址、API Key
- 合約內容、報價金額、內部成本結構
- 客戶個人資料（姓名、身份證字號、聯絡方式）
- 任何可直接辨識個人或組織的機密資訊
- 含有帳號資訊、系統網址、內部資料表、客戶名稱、訂單編號、交易金額的系統截圖

**A 級資料不得寫入原文、截圖、數值、識別資訊或可反推內容；任務包中僅可記錄「已排除 A 級敏感資料」或「本資料因敏感性不納入分析」等抽象說明。**

**B 級 — 可摘要，不可原文寫入**

以下資料可以摘要後引用，但不得原文複製貼入：

- 客戶提供的流程文件、現行 SOP、操作手冊
- 訪談逐字稿、會議錄音轉錄
- 客戶內部簽核規則、授權層級說明
- 具體金額數字、人員職稱、部門名稱

> 使用方式：以結構化方式描述，例如「企業目前採購申請流程為三級簽核」，不貼入原始文字

**C 級 — 整理後可寫入任務包**

以下資料經整理、結構化後，可寫入對應任務包文件：

- 企業流程節點（已去除個人資訊後的流程步驟）
- 系統設定需求（以功能描述而非操作截圖方式記錄）
- 待確認事項（以問題描述方式記錄，不含敏感原始資料）
- 顧問觀察與分析判斷

## 七、正式交付成果存放規則（例外流程）

**預設**：所有產出存放於 `91_跨員工協作區/[任務包]/`，不自動寫入 90。

只有在以下條件**全部成立**時，方可將產出移入 `90_正式交付成果/[企業代號]/`：

| 條件 | 確認狀態 |
|------|----------|
| 使用者明確指示「本次產出為正式交付成果」 | ⬜ / ✅ |
| 產出已完成品質確認，非草稿 | ⬜ / ✅ |
| 企業代號已於 90 下建立對應子資料夾或可建立 | ⬜ / ✅ |
| 文件不包含 A 級資料，B 級資料已完成摘要處理 | ⬜ / ✅ |

## 八、知識沉澱預設規則

任務完成後**必須執行知識沉澱判斷**，但不代表必須寫入知識庫。

**寫入 02_通用流程知識庫的五項條件（全部符合才可寫入）**：

| 條件 | 說明 |
|------|------|
| ① 真實任務 | 來源為真實企業任務包，Test_* 測試任務包內容不得收錄 |
| ② 已脫敏 | 所有企業識別資訊已依第五區塊規則完成抽象化處理 |
| ③ 可跨企業複用 | 此知識對其他企業同類型流程有參考價值 |
| ④ 非客戶專屬規則 | 不是客戶特定的業務規則、金額標準或組織規定 |
| ⑤ 經使用者確認 | 在對話中呈現草稿，使用者確認後方可寫入 |

若不符合五項條件，判斷結論為「本次無可沉澱知識」，記錄於 worklog 即可。

## 九、檢查表完成後的下一步

完成本檢查表後，依以下順序執行：

1. **輸出任務包命名建議**：依第一區塊資訊，提出正式任務包資料夾名稱
2. **確認啟用模板清單**：依第三區塊勾選結果，列出本次需複製的模板
3. **詢問使用者是否同意建立任務包**：明確等待使用者回覆，不得預先建立
4. **使用者確認後才可執行**：建立任務包資料夾、複製對應模板、填寫 00_任務說明.md 基本資訊
5. **將本檢查表摘要寫入 00_任務說明.md**，至少包含：
   - 企業代號 / 任務名稱
   - 任務範圍與不處理事項
   - 預期產出與完成標準
   - 輸入資料摘要與敏感等級判斷
   - 本次啟用模板
   - 是否允許寫入 90
   - 任務完成後是否啟動知識沉澱判斷
