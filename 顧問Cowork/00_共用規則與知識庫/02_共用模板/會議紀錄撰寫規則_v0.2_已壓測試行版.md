## 一、章節結構規則

### 1.1 固定結構順序

ERP 輔導備忘錄由四個固定區塊組成，順序固定，【會後補充】為條件選填：

```
[區塊 A] 標頭資訊（必填）
[區塊 B] 討論內容（必填）
[區塊 C] 決議事項（必填，若無決議填固定說明句）
[區塊 D] 待辦追蹤（建議必填，確認無待辦可省略）
[條件選填] 【會後補充】（非決議、非待辦的背景補充說明才使用）
```

### 1.2 各區塊說明

| 區塊 | 名稱 | 必填 / 選填 | 說明 |
|------|------|------------|------|
| A | 標頭資訊 | 必填 | 會議基本資料，含主題、客戶、日期、人員、顧問、時數 |
| B | 討論內容 | 必填 | 會議主體，條列式紀錄所有討論事項 |
| C | 【決議事項】 | 必填（無決議填固定說明句）| 記錄本次會議形成之明確決議，使用 D-001 編號；不得省略 |
| D | 【待辦追蹤】 | 建議必填 | 列出後續行動項目，使用 TO-001 表格格式；確認無待辦可省略 |
| — | 【會後補充】 | 條件選填 | 僅用於背景補充、名詞解釋、系統前提說明等非決議、非待辦內容；不得取代 C 或 D 區塊 |

> **原則：** 決議與待辦必須分開區塊，不得混入「討論內容」主體。【決議事項】區塊不得省略。

## 二、欄位設計規則

### 2.1 標頭欄位（區塊 A）

| 欄位名稱 | 欄位類型 | 必填條件 | 備註 |
|----------|----------|----------|------|
| 會議主題 | 文字 | 必填 | 簡短描述本次討論主題，10～30 字為宜 |
| 客戶名稱 | 文字 | 必填 | 客戶正式公司名稱；資料不足時填 `[待確認：客戶全名]` |
| 會議日期 | 日期 + 時間範圍 | 必填 | 格式：`YYYY/MM/DD HH:MM～HH:MM` |
| 參與人員 | 文字（三層格式） | 必填 | 格式：`公司別 / 部門 / 姓名`；缺少資訊時於缺少層標示 `[待確認]`，詳見 2.1.1 |
| 執行顧問 | 文字 | 必填 | 主責顧問姓名 |
| 輔導時數 | 數字 + H | **計費型必填；非計費型填 `不適用`；無法確認填 `[待確認]`** | 格式：`X.XH`，如 `1.5H` |
| 交通時數 | 數字 + H | **計費型必填；非計費型填 `不適用`；無法確認填 `[待確認]`** | 無交通時填 `0H`，不得留空（計費型） |

#### 2.1.1 參與人員填寫格式

格式：`公司別 / 部門 / 姓名`

```
範例（完整資訊）：
  艾創點數位 / ERP 顧問部 / 王小明
  建大工業 / 資訊部 / 陳大文

範例（部分資訊缺少）：
  艾創點數位 / ERP 顧問部 / [待確認：顧問姓名]
  建大工業 / [待確認：部門] / [待確認：窗口姓名]

範例（資訊完全不明）：
  [待確認：客戶方出席人員]
```

### 2.2 討論內容欄位（區塊 B）

採多層條列編號，不使用自由段落。

| 層級 | 格式 | 用途 |
|------|------|------|
| 第一層 | `1.` | 主要議題 / 討論主題 |
| 第二層 | `1.1.` | 該議題的具體說明 / 結論 |
| 第三層 | `1.1.1.` | 進一步細節、條件說明或例外情境 |

### 2.3 決議事項欄位（區塊 C）

使用 `D-001`、`D-002` 獨立編號，每條決議獨立一行。

```
【決議事項】

D-001：（決議內容，記述式，不帶主觀語氣）
D-002：（決議內容）或 [待確認：是否已形成正式決議]
```

**無決議時固定填入：**
```
本次會議未形成明確決議，相關事項列入【待辦追蹤】或標示為 [待確認]。
```

> 不得省略【決議事項】區塊，有無決議均需填寫。

### 2.4 待辦追蹤欄位（區塊 D）

採表格格式，使用 `TO-001`、`TO-002` 獨立編號。

```
【待辦追蹤】

| 編號   | 責任方       | 待辦事項                    | 期限       | 狀態  |
|--------|-------------|----------------------------|------------|-------|
| TO-001 | (客戶窗口)  | （可執行的動詞句）             | [待確認]   | 待處理 |
| TO-002 | (顧問端)    | （待辦事項描述）               | YYYY/MM/DD | 待處理 |
```

**欄位說明：**

| 欄位 | 說明 |
|------|------|
| 編號 | TO-001 起跳，每份備忘錄獨立計；跨紀錄追蹤時可引用編號，如「追蹤 TO-003 進度」 |
| 責任方 | 以 `(客戶窗口)`、`(顧問端)`、`(雙方)` 標示；姓名不明時用職稱或部門 |
| 待辦事項 | 必須為可執行的動詞句，不可只列名詞 |
| 期限 | 有承諾日期填 `YYYY/MM/DD`；未確認填 `[待確認]` |
| 狀態 | 預設「待處理」；後續追蹤時更新為「進行中」/ 「已完成」/ 「逾期」 |

> v0.1 的主文流水號接續方式（5、6、7…）正式廢止。

## 三、填寫規範

### 3.1 日期與時間

- 日期格式：`YYYY/MM/DD`，不可簡寫為 `MM/DD` 或 `YY/MM/DD`
- 時間範圍格式：`HH:MM～HH:MM`（使用全形波浪號 `～`）
- 輔導時數精確到 0.5H，不記分鐘

### 3.2 時數欄位適用範圍

| 場景 | 輔導時數 | 交通時數 |
|------|----------|----------|
| ERP 顧問輔導（計費型） | **必填**，如 `1.5H` | **必填**，無交通填 `0H` |
| 非計費型會議 / 內部討論 | 填 `不適用` | 填 `不適用` |
| 無法確認是否計費型 | 填 `[待確認]` | 填 `[待確認]` |

> 時數欄位不得刪除，保留欄位以維持格式穩定。

### 3.3 待辦項目編號規則

**v0.2 正式採用 `TO-001` 獨立編號，廢止 v0.1 的主文流水號接續方式。**

- 每份備忘錄的待辦項目從 `TO-001` 起跳，不延續上份紀錄
- 跨紀錄追蹤時，直接引用編號，如「追蹤上次 TO-003 進度」
- 同一份備忘錄不得出現相同 TO 編號

### 3.4 特殊標記規則

| 標記 | 用途 | 範例 |
|------|------|------|
| `【 】` 全形括號 | 區塊標題 | `【決議事項】`、`【待辦追蹤】`、`【會後補充-主題說明】` |
| `( )` 圓括號 | 責任歸屬 | `(客戶端)`、`(顧問端)`、`(雙方)` |
| `[ ]` 方括號 | 尚待確認的選項 | `[系統/人工]`、`[待確認]`、`[待確認：姓名]` |
| `『 』` 書名號 | 引用文件 / 功能名稱 | `『現行作業規範』` |
| `D-001` | 決議事項編號 | `D-001：確認採用 X 方案` |
| `TO-001` | 待辦事項編號 | `TO-001` 欄位第一格 |

### 3.5 必填與留空規則

- 客戶名稱確認前可用佔位符，但正式版不得留空
- 【決議事項】不得省略：有決議填 D-001；無決議填固定說明句
- 【待辦追蹤】有待辦就必須列出；確認無待辦可省略
- 技術設定、API 參數若尚未確認，一律標記 `[待確認]`，不得臆測
- 時數欄位不得刪除；計費型填時數，非計費型填 `不適用`

### 3.6 禁止事項

- 禁止在討論內容中直接引用未確認的技術參數（應標記 `[待確認]`）
- 禁止在討論主體中夾帶待辦事項（待辦必須移至 `【待辦追蹤】`）
- 禁止在討論主體中夾帶決議（決議必須移至 `【決議事項】`）
- 禁止多個議題共用一個條列號碼（每個議題獨立編號）
- 禁止省略【決議事項】區塊，即使無決議也需填固定說明句
- 禁止用【會後補充】區塊替代【決議事項】或【待辦追蹤】

### 3.7 【決議事項】填寫規範

| 情境 | 填寫方式 |
|------|---------|
| 有明確決議 | `D-001：（決議內容，記述式）` |
| 決議尚未確認 | `D-001：[待確認：本項是否已形成正式決議]` |
| 完全無決議 | 固定說明句：「本次會議未形成明確決議，相關事項列入【待辦追蹤】或標示為 [待確認]。」 |

> 決議內容應以記述式呈現（「確認採用 X 方案」），不以主觀推測填寫（「感覺大家同意 X」）。

## 四、顧問語氣與風格

### 4.1 核心語氣定義

**技術說明型、中立記述、不帶主觀評論。**

| 特徵 | 說明 | 例子 |
|------|------|------|
| 動詞偏技術說明用語 | 採取、依照、確認、建議 | 「通訊協定採取 RESTful API」 |
| 記述而非評論 | 只記錄決議與說明，不加顧問個人意見 | 「建議線下協議」（客觀轉述），而非「應該要線下處理比較好」 |
| 含標題性說明標記 | 使用 `【說明】` 標記解釋性補充 | `【說明】採購訂單對接流程` |
| 條件性選項明確標記 | 不確定事項用 `[ ]` 明示 | `[系統/人工]` |

### 4.2 中英混用原則

- 技術術語、API 名稱、欄位名稱允許英文
- 一般描述用中文，英文縮寫首次出現需說明
- 中英混用時不加引號，直接嵌入句子中

### 4.3 語氣禁止清單

- 禁止使用「我覺得」、「可能」、「也許」等模糊主觀詞
- 禁止使用「非常」、「超級」等誇飾詞
- 禁止用過去式被動語態報告（「已被確認」→ 改為「已確認」）
- 不確定事項應標記 `[待確認]`，而非用模糊語氣帶過

## 五、資訊呈現方式

### 5.1 整體版型

採「表格式標頭 + 條列式主體 + 決議事項 + 待辦追蹤」四段式版型：

```
┌─────────────────────────────────────────────────┐
│  標頭資訊（表格 / 對齊欄位）                      │
│  公司別/部門/姓名 | 日期時間 | 顧問 | 時數         │
└─────────────────────────────────────────────────┘

討論內容
  1. 第一議題
    1.1. 說明 / 結論
    1.2. ...
  2. 第二議題
    ...

【決議事項】（必填，無決議填固定說明句）
  D-001：（決議內容）
  D-002：...

【會後補充-主題說明】（條件選填，非決議非待辦才使用）
  說明或補充內容

【待辦追蹤】（建議必填）
  | 編號   | 責任方 | 待辦事項 | 期限 | 狀態 |
  | TO-001 | ...   | ...     | ... | ... |
```

### 5.2 標頭呈現

- 標頭欄位採橫向對齊排列，相關欄位同行（如：客戶名稱 + 會議日期同一行）
- 輔導時數與交通時數同行
- 參與人員採三層格式（公司別 / 部門 / 姓名），各層以 ` / ` 分隔

### 5.3 討論內容呈現

- 多層縮排條列，最多三層
- 引用的文件 / 功能名稱用書名號 `『 』` 標示
- 技術術語 / API 名稱保留英文原文，不硬翻
- 每個議題間留一個空行，提升可讀性

### 5.4 待辦追蹤呈現

- 採表格格式，欄位：編號 / 責任方 / 待辦事項 / 期限 / 狀態
- TO 編號從 TO-001 起跳，每份備忘錄重新計
- 待辦項目每條獨立一列，不合併
- 期限未確認填 `[待確認]`，有承諾日期填 `YYYY/MM/DD`

### 5.5 決議事項呈現

- 【決議事項】區塊標題單獨一行，不接文字
- D 編號從 D-001 起跳，每份備忘錄重新計
- 每條決議獨立一行，不合併
- 無決議時填固定說明句，不省略區塊
- 決議內容不得帶主觀推測語氣

## 六、品質標準

### 6.1 必備品質條件（全部達成才可交付）

| 條件 | 說明 |
|------|------|
| 標頭完整 | 必填欄位均已填寫（時數欄視場景而定），無遺漏 |
| 議題有結論 | 每個討論議題至少有一個明確結論、決議或 `[待確認]` 標記 |
| 決議區塊已填寫 | 有決議填 D-001；無決議填固定說明句；不得省略【決議事項】區塊 |
| 待辦有責任方 | 每條待辦項目均標示負責方，不得只列事項不列責任 |
| 時數已記錄 | 計費型：輔導時數與交通時數均已填寫；非計費型：填 `不適用` |
| 無混淆標記 | `[待確認]` / D-001 / TO-001 / `【】` 等標記使用一致，不混用 |

### 6.2 建議品質條件（提升可追蹤性）

| 條件 | 說明 |
|------|------|
| 引用文件名稱明確 | 提到相關文件時使用 `『文件名稱』` 格式，不用「那份文件」 |
| 技術設定有來源 | 涉及系統設定、API 參數時，標明依據文件或確認方 |
| 待辦有期限 | TO 表格的期限欄填入承諾日期或 `[待確認]`，不留空 |
| 分段清晰 | 討論內容、決議事項、待辦追蹤各自獨立，不混入 |

### 6.3 不可接受的品質問題

- 待辦項目有事沒有責任方
- 討論議題沒有任何結論或待確認標記
- 客戶名稱、顧問姓名為空白（草稿可用佔位符，正式版不行）
- `【】` 標記混用（如將 `【說明】` 誤用為 `【待辦追蹤】` 區塊）
- 省略【決議事項】區塊
- 待辦使用流水號接續方式（應使用 TO-001 表格）
