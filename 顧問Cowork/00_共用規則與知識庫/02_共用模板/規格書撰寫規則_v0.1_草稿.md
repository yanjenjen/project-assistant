## 一、適用限制

1. 本規則由兩份公司範例模板萃取後整合，代表目前可觀察到的撰寫慣例，**不代表唯一正確做法**。
2. 適用範圍：Odoo ERP 系統的**標準作業模組規格書**與**API 介接規格書**。
3. **不適用**以下情境：
   - 非 Odoo 系統的規格設計
   - 操作說明手冊（屬 B2 範本，另有規則）
   - 驗收文件、交付清單等其他文件類型
4. 若實際專案需求與本規則衝突，以專案確認版本為準，並在規格書中標示差異。
5. **若需作為正式共用撰寫規則，應再蒐集更多範本樣本進行比對驗證。**

## 二、規格書類型判斷表

| 判斷問題 | 標準作業規格書（範本 01） | API 作業規格書（範本 02） |
|---|---|---|
| 是否涉及 Odoo 系統畫面設計（TreeView、FormView 等）？ | ✅ 是 | ❌ 否 |
| 是否涉及外部系統透過 API 與 Odoo 交換資料？ | ❌ 否 | ✅ 是 |
| 是否需要設計權限群組、搜尋、篩選、分組？ | ✅ 是 | ❌ 否 |
| 是否需要定義 Endpoint、HTTP Method、Request / Response？ | ❌ 否 | ✅ 是 |
| 主要交付對象是否為 Odoo 開發工程師（畫面與功能）？ | ✅ 是 | 部分 |
| 主要交付對象是否為介接方開發工程師？ | ❌ 否 | ✅ 是 |

> 若同一需求涉及兩種類型（例如：外部系統透過 API 傳入資料後，Odoo 端需設計對應作業畫面），應分開撰寫兩份文件，不可合併。

## 三、標準作業規格書撰寫規則

### 3.1 章節結構

文件固定五大章節，依序排列，不可省略或調換順序：

| # | 章節名稱 | 說明 |
|---|---|---|
| 一 | 【需求說明】 | 業務需求起因，條列式，1 至數條 |
| 二 | 【作業功能】 | 含四個固定子節：權限、搜尋、篩選、分組 |
| 三 | 【作業畫面】 | 含欄位型態參考表 + 各 View 設計 |
| 四 | 【列表（TreeView）Action】 | 快速按鈕、動作按鈕、列印 |
| 五 | 【表單（FormView）Action】 | 快速按鈕、智慧按鈕、動作按鈕、列印 |

文件開頭應包含**自動目錄**，完成後手動更新。

### 3.2 第二章子節規則

**2.1 權限群組規劃**
- 全新作業：必須填寫「存取權」與「紀錄規則」兩張表
- 既有功能增加：可標示「略」，不需填寫完整表格

**2.2 搜尋功能**
- 每個搜尋條件需標明：搜尋名稱、資料來源（單頭 / 單身）、額外處理邏輯
- **Selection 與 Boolean 型態欄位需特別注意**：是否可作為搜尋條件，應依實際 Odoo Search View 設計方式確認，不可直接視為通用限制或保證支援。

**2.3 篩選功能**
- 每個篩選條件需標明：是否預設啟用、組別編號
- 同一組別內的條件為 OR 關係；不同組別為 AND 關係

**2.4 分組功能**
- 標明分組來源與是否預設啟用

### 3.3 第三章欄位設計規則

**欄位型態代號表（第三章開頭必附）：**

| 代號 | 型態說明 |
|---|---|
| C1 | Char（單行文字） |
| C2 | Text（多行文字） |
| N1 | Integer（整數） |
| N2 | Float（浮點數） |
| D1 | Date（日期） |
| D2 | Datetime（日期+時間） |
| B | Boolean（勾選） |
| S1 | Selection（下拉單選） |
| S2 | Many2one（關聯單選） |
| S3 | Many2many（關聯多選） |
| H | Html（圖文編輯器） |
| P | Image（圖片） |
| F | Binary（附件） |

**資料結構主表格欄位：**
- 欄位說明、欄位代號、型態、唯讀、必輸、追蹤、資料處理（預設邏輯）
- **「資料處理」欄不得空白**；若無特殊邏輯，須明確填寫「無特殊處理」或「依 Odoo 標準行為」，避免工程師誤解為待補欄位。

**View 選用建議（非強制）：**
- 基礎資料（如幣別）：規劃 TreeView 即可
- 主檔資料（如客戶、料件）：TreeView + FormView，可加 KanbanView
- 表單資料（如訂單）：同主檔，可加 CalendarView 做日期篩選

**TreeView 自訂顯示規則：**
- 空白 = 永遠顯示，不可隱藏
- `show` = 預設顯示，可由使用者隱藏
- `hide` = 預設隱藏，可由使用者顯示

### 3.4 截圖規則

| 文件版本 | 截圖要求 |
|---|---|
| **規格初稿** | 可使用示意圖或 wireframe，明確標示「示意圖，待確認」 |
| **確認版** | 應替換為對應作業的設計圖或經顧問與客戶確認的示意圖 |
| **交付版** | 應使用實際畫面截圖或最終確認版設計圖，不得留有初稿示意圖 |

### 3.5 Action 區塊規則

- 快速按鈕、動作按鈕（齒輪）、智慧按鈕、列印各為獨立子節
- 「刪除」「複製」為 Odoo 標準功能，不需特別列出
- 「封存」**非**標準功能，須在規格中明確提出

## 四、API 作業規格書撰寫規則

### 4.1 章節結構

文件固定兩大章節：

| # | 章節名稱 | 說明 |
|---|---|---|
| 一 | 【文件說明】 | 環境網址、API 清單總覽、注意事項 |
| 二 | 【API 清單】 | 每個 API 一個子節 |

**第一章固定子節：**
- **API 介接網址**：正式環境與測試環境各一行
- **API 清單總覽＆說明**：彙整所有 API，欄位：名稱、性質（新增 / 刪除 / 查詢 / 修改）、簡述
- **注意事項**：跨 API 通用規則，如認證機制、KEY 取得方式、使用前提

**第二章子節命名規則：**
動詞 + 對象，例如：「取得 XX 金鑰」「新增 XX 單據」「刪除 XX 單據」「查詢 XX 資料」

### 4.2 每個 API 子節的固定區塊

| 區塊 | 必要 / 選填 | 填寫規則 |
|---|---|---|
| Endpoint | 必要 | 小寫路徑，動詞為英文（`/new`、`/delete`、`/search`） |
| Method | 必要 | POST / GET / DELETE / PUT，依語義選擇 |
| 說明 | 必要 | 「1. 動詞 + 業務目的」格式，可多條 |
| Header | 必要（不可省略） | 若需認證：填寫認證方式（如 `Authorization: APIKEY`）；**若不需認證：明確標示「無」或「不需認證」**，不可空白 |
| Body | 有請求參數時必要 | 使用多欄表格（見 4.3） |
| 狀態碼 | 必要 | 列出所有可能的狀態碼與對應說明；**不可預設只有 200 / 500**，應依實際設計列出 |
| 回傳 | 必要 | 使用多欄表格（見 4.3） |
| 回傳\_data | data 物件存在時必要 | 列出 data 物件內的欄位 |
| Example | 必要 | Request + Response 各一個 JSON 範例（使用示意值，不含真實資料） |
| Odoo 欄位對應 | 必要 | 使用對應表格（見 4.3） |

### 4.3 表格欄位規則

**Body / 回傳表格：**

| Column Name（英文） | 欄位名稱（中文） | 型態 | 備註（必填標示 `*資料必填`、格式、允許值） |

**型態標示規則：**
- 預設使用英文小寫：`string`、`bool`、`integer`、`numeric`、`datetime`
- 日期時間格式統一為 `%Y-%m-%d %H:%M:%S`，必須在備註欄明確標示
- 同一份文件型態寫法必須一致，不可混用大小寫

**Odoo 欄位對應表格：**

| Column Name | 欄位名稱 | 型態 | 欄位代號（model.field 格式） | 條件 |

- 欄位代號使用完整路徑，例如：`sale_order.partner_id.name`
- **若欄位對應尚未確認**：填寫「待確認」或說明「需依轉換邏輯確認」，不可空白
- 條件欄：有查詢條件限制時必須填入；無條件則留空（不填「無」）

### 4.4 Example 規則

- Request / Response 使用標準 JSON 格式，縮排一致，字串值使用雙引號
- 範例中的資料值為**示意值**，不得使用真實環境帳號、密碼、客戶名稱、金額
- Response 標準結構固定包含：`success`（bool）、`message`（string）、`data`（object 或 null）

## 五、兩種規格書共通規則

| 規則 | 說明 |
|---|---|
| **自動目錄** | 文件開頭放自動目錄，完成後手動更新 |
| **章節標題格式** | 使用【】標示主章節名稱，為固定慣例 |
| **欄位技術代號與中文說明** | 欄位需具備可供工程師辨識的技術代號或英文欄位名稱，並搭配中文說明；標準作業規格書以欄位代號與型態為主，API 規格書以 Column Name 與中文欄位名稱對照為主 |
| **表格為主要呈現方式** | 功能設計、欄位定義、對應關係均使用表格，不用純文字段落 |
| **TIP 區塊交付前必須清除** | 標準作業規格書中的黃色 TIP 說明文字為內部提示，交付客戶前必須全部刪除 |
| **必填標示** | 必填欄位在備註欄標示 `*資料必填` |
| **「略」的使用** | 若某子節或功能不適用，明確標示「略」，不留空白 |
| **顧問編輯區標示** | 綠色字體為顧問填寫區，正式交付前應確認所有綠色區塊均已填寫完整 |

## 六、兩種規格書差異表

| 比較維度 | 標準作業規格書（範本 01） | API 作業規格書（範本 02） |
|---|---|---|
| **主要目的** | 設計 Odoo 系統畫面與功能邏輯 | 定義外部系統與 Odoo 的 API 介接規格 |
| **章節數量** | 五大章節 | 兩大章節 |
| **欄位型態表示** | 代號（C1、N2、D1…） | 英文型態名（string、integer…） |
| **Odoo 欄位深度** | 抽象層（欄位代號、必輸 / 唯讀屬性） | 具體技術層（完整 model.field 路徑） |
| **截圖** | 必要，每個 View 附截圖 | 無截圖 |
| **TIP 說明** | 大量，交付前須清除 | 極少 |
| **Example** | 無 JSON 範例 | 必須附 Request + Response |
| **認證設計** | 無（由 Odoo 權限群組規劃處理） | 必須說明（Header / APIKEY 等） |
| **狀態碼** | 無 | 必須列出全部可能值 |
| **主要讀者** | Odoo 開發工程師（畫面端） | 介接方開發工程師 |
