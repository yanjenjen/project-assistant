## 一、章節結構規則

### 1.1 固定結構順序

ERP 輔導備忘錄由三個區塊組成，順序固定：

```
[區塊 A] 標頭資訊（必填）
[區塊 B] 討論內容（必填）
[區塊 C] 會後補充 / 待辦追蹤區塊（選填，依需要加入）
```

### 1.2 各區塊說明

| 區塊 | 名稱 | 必填 / 選填 | 說明 |
|------|------|------------|------|
| A | 標頭資訊 | 必填 | 會議基本資料，含主題、客戶、日期、人員、顧問、時數 |
| B | 討論內容 | 必填 | 會議主體，條列式紀錄所有討論事項 |
| C-1 | 【會後補充】 | 選填 | 會議後延伸的確認事項或下一步規劃，用【】標示 |
| C-2 | 【待辦追蹤】 | 建議填寫 | 列出負責方與待辦事項，用【】標示 |

> **原則：** 若有會後補充或待辦，必須分開區塊，不得混入「討論內容」主體。

## 二、欄位設計規則

### 2.1 標頭欄位（區塊 A）

| 欄位名稱 | 欄位類型 | 必填條件 | 備註 |
|----------|----------|----------|------|
| 會議主題 | 文字 | 必填 | 簡短描述本次討論主題，10～30 字為宜 |
| 客戶名稱 | 文字 | 必填 | 客戶正式公司名稱；草稿階段可用「XX 股份有限公司」佔位 |
| 會議日期 | 日期 + 時間範圍 | 必填 | 格式：`YYYY/MM/DD HH:MM～HH:MM` |
| 參與人員 | 文字（逗號分隔） | 必填 | 記錄部門或姓名；若資料不足以部門代替 |
| 執行顧問 | 文字 | 必填 | 主責顧問姓名 |
| 輔導時數 | 數字 + H | **ERP 輔導場景必填；非計費型會議可選填或省略** | 格式：`X.XH`，如 `1.5H` |
| 交通時數 | 數字 + H | **ERP 輔導場景必填；非計費型會議可選填或省略** | 無交通時記錄 `0H`，不得留空（ERP 輔導場景） |

### 2.2 討論內容欄位（區塊 B）

採多層條列編號，不使用自由段落。

| 層級 | 格式 | 用途 |
|------|------|------|
| 第一層 | `1.` | 主要議題 / 討論主題 |
| 第二層 | `1.1.` | 該議題的具體說明 / 結論 |
| 第三層 | `1.1.1.` | 進一步細節、條件說明或例外情境 |

### 2.3 會後補充 / 待辦追蹤區塊欄位（區塊 C）

**【待辦追蹤】格式：**

```
N. (負責方) 待辦事項描述
```

- `N.` 為待辦項目編號（詳見填寫規範 3.3）
- `(負責方)` 用圓括號標示，可為客戶方或顧問方
- 待辦事項描述需為可執行的動詞句

## 三、填寫規範

### 3.1 日期與時間

- 日期格式：`YYYY/MM/DD`，不可簡寫為 `MM/DD` 或 `YY/MM/DD`
- 時間範圍格式：`HH:MM～HH:MM`（使用全形波浪號 `～`）
- 輔導時數精確到 0.5H，不記分鐘

### 3.2 時數欄位適用範圍

| 場景 | 輔導時數 | 交通時數 |
|------|----------|----------|
| ERP 顧問輔導（計費型） | **必填**，無交通填 `0H` | **必填**，無交通填 `0H` |
| 非計費型會議 / 內部討論 | 可選填或省略 | 可選填或省略 |

> 若省略時數欄位，需在標頭說明：「本份紀錄不含時數紀錄」。

### 3.3 待辦項目編號規則

**D2 原始格式說明：** 原範本中，待辦項目編號可能接續討論內容的流水號（如討論內容為 1～4，待辦為 5、6、7），此為早期備忘錄的常見寫法。

**v0.1 規則：** 可保留接續流水號的方式，以維持與現有範本的一致性，便於觀察實際使用情形。

**v0.2 建議方向：** 應優先改為**獨立待辦編號**（如 `TO-001`、`TO-002`），原因如下：
- 獨立編號可跨紀錄追蹤待辦狀態
- 不受討論內容數量影響，編號穩定
- 便於後續建立待辦清單或轉入任務管理工具

### 3.4 特殊標記規則

| 標記 | 用途 | 範例 |
|------|------|------|
| `【 】` 全形括號 | 會後補充 / 待辦追蹤區塊標題 | `【待辦追蹤】`、`【會後補充-主題說明】` |
| `( )` 圓括號 | 責任歸屬 | `(客戶方)`、`(顧問方)` |
| `[ ]` 方括號 | 尚待確認的選項 | `[系統/人工]`、`[待確認]` |
| `『 』` 書名號 | 引用文件 / 功能名稱 | `『現行作業規範』` |

### 3.5 必填與留空規則

- 客戶名稱確認前可用佔位符，但正式版不得留空
- 【待辦追蹤】若無待辦，可省略此區塊；有待辦就必須列出
- 技術設定、API 參數若尚未確認，一律標記 `[待確認]`，不得臆測

### 3.6 禁止事項

- 禁止在討論內容中直接引用未確認的技術參數（應標記 `[待確認]`）
- 禁止在討論主體中夾帶待辦事項（待辦必須移至 `【待辦追蹤】`）
- 禁止多個議題共用一個條列號碼（每個議題獨立編號）

## 四、顧問語氣與風格

### 4.1 核心語氣定義

**技術說明型、中立記述、不帶主觀評論。**

| 特徵 | 說明 | 例子 |
|------|------|------|
| 動詞偏技術說明用語 | 採取、依照、確認、建議 | 「通訊協定採取 RESTful API」 |
| 記述而非評論 | 只記錄決議與說明，不加顧問個人意見 | 「建議線下協議」（客觀轉述），而非「應該要線下處理比較好」 |
| 含標題性說明標記 | 使用 `【說明】` 標記解釋性補充 | `【說明】採購訂單對接流程` |
| 條件性選項明確標記 | 不確定事項用 `[ ]` 明示 | `[系統/人工]` |

### 4.2 中英混用原則

- 技術術語、API 名稱、欄位名稱允許英文
- 一般描述用中文，英文縮寫首次出現需說明
- 中英混用時不加引號，直接嵌入句子中

### 4.3 語氣禁止清單

- 禁止使用「我覺得」、「可能」、「也許」等模糊主觀詞
- 禁止使用「非常」、「超級」等誇飾詞
- 禁止用過去式被動語態報告（「已被確認」→ 改為「已確認」）
- 不確定事項應標記 `[待確認]`，而非用模糊語氣帶過

## 五、資訊呈現方式

### 5.1 整體版型

採「表格式標頭 + 條列式主體 + 會後補充 / 待辦追蹤」三段式版型：

```
┌─────────────────────────────────────────┐
│  標頭資訊（表格 / 對齊欄位）              │
└─────────────────────────────────────────┘

討論內容
  1. 第一議題
    1.1. 說明 / 結論
    1.2. ...
  2. 第二議題
    ...

【會後補充-主題說明】（選填）
  說明或補充內容

【待辦追蹤】（建議填寫）
  N. (責任方) 待辦事項
```

### 5.2 標頭呈現

- 標頭欄位採橫向對齊排列，相關欄位同行（如：客戶名稱 + 會議日期同一行）
- 輔導時數與交通時數同行（ERP 輔導場景）
- 非計費場景省略時數欄時，標頭仍需保留其他必填欄位

### 5.3 討論內容呈現

- 多層縮排條列，最多三層
- 引用的文件 / 功能名稱用書名號 `『 』` 標示
- 技術術語 / API 名稱保留英文原文，不硬翻
- 每個議題間留一個空行，提升可讀性

### 5.4 會後補充 / 待辦追蹤呈現

- `【】` 區塊標題單獨一行，不接文字
- 區塊間留空行分隔
- 待辦項目每條獨立一行，不合併
- 【會後補充】與【待辦追蹤】為兩個獨立區塊，不合併使用

## 六、品質標準

### 6.1 必備品質條件（全部達成才可交付）

| 條件 | 說明 |
|------|------|
| 標頭完整 | 必填欄位均已填寫（時數欄視場景而定），無遺漏 |
| 議題有結論 | 每個討論議題至少有一個明確結論、決議或 `[待確認]` 標記 |
| 待辦有責任方 | 每條待辦項目均標示負責方，不得只列事項不列責任 |
| 時數已記錄 | ERP 輔導場景：輔導時數與交通時數均已填寫；非計費場景：已說明省略原因 |
| 無混淆標記 | `[待確認]` / `【待辦追蹤】` / `[系統/人工]` 等標記使用一致，不混用 |

### 6.2 建議品質條件（提升可追蹤性）

| 條件 | 說明 |
|------|------|
| 引用文件名稱明確 | 提到相關文件時使用 `『文件名稱』` 格式，不用「那份文件」 |
| 技術設定有來源 | 涉及系統設定、API 參數時，標明依據文件或確認方 |
| 待辦有時限 | 【待辦追蹤】若可行，標明預計完成時間 |
| 分段清晰 | 討論內容、會後補充、待辦追蹤各自獨立，不混入 |

### 6.3 不可接受的品質問題

- 待辦項目有事沒有責任方
- 討論議題沒有任何結論或待確認標記
- 客戶名稱、顧問姓名為空白（草稿可用佔位符，正式版不行）
- `【】` 標記混用（如將 `【說明】` 誤用為 `【待辦追蹤】` 區塊）

## 八、目前限制與後續補強建議

### 8.1 本版限制

| 限制 | 說明 |
|------|------|
| 僅基於單一範本 | 只有 D2 輔導備忘錄一份來源，格式偏向技術整合型討論，尚未涵蓋需求訪談、功能確認、上線前準備等場景 |
| 無決議追蹤機制 | 決議混在討論內容中，不利後續追蹤 |
| 待辦追蹤格式較弱 | 待辦項目 v0.1 暫用接續流水號，無截止日欄位、無狀態欄位 |
| 無範本壓測 | 本規則尚未用於實際會議紀錄產出壓測 |

### 8.2 後續補強建議（v0.2 方向）

| 優先度 | 補強項目 |
|--------|----------|
| 🔴 高 | 加入【決議事項】獨立區塊，明確記錄每條決議的責任方與期限 |
| 🔴 高 | 待辦項目改用獨立流水號（`TO-001`），並加入截止日與狀態欄 |
| 🟠 中 | 增加「需求訪談型」場景範例，補充訪談問題 / 客戶回答的記錄格式 |
| 🟠 中 | 建立壓測任務包，用本規則產出一份模擬會議紀錄，驗證格式可用性 |
| 🟡 低 | 評估是否建立 `meeting-minutes-formatter` Skill（待本規則壓測通過後再決定） |
