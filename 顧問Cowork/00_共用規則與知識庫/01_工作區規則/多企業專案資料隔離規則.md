## 1. 文件目的

本文件用於規範「顧問Cowork」工作區在同時服務多家企業或多個 ERP 導入專案時，應遵守的資料隔離原則、資料夾結構規則、檔案命名規範與任務啟動前的確認機制。

本文件的存在目的是防止以下風險：

- 不同企業的流程資料、分析結論或交付文件互相混入
- 未指定服務對象即開始建立正式檔案
- 多專案並行時資料夾歸屬不清、版本混亂
- 草稿與正式交付成果放置位置錯誤

本文件為工作區共用規則，適用於三位 AI 員工（員工_01、員工_02、員工_03）的所有任務執行行為，以及使用者對工作區的所有操作。

---

## 2. 適用範圍

本規則適用於以下所有情境：

| 情境 | 說明 |
|------|------|
| 單企業單專案 | 僅服務一家企業，但仍需依規則建立命名與資料夾結構，確保後續可擴充 |
| 單企業多專案 | 同一企業有多個 ERP 模組或階段並行，需依專案或模組分開管理 |
| 多企業並行 | 同時服務兩家以上企業，為最高風險情境，本規則最為關鍵 |
| 歷史專案查閱 | 查閱已完成專案的資料時，同樣適用資料夾與命名規則，不得將歷史資料搬移至其他企業 |

## 3. 核心原則

### 3.1 每次任務必須指定企業 / 專案名稱

任務開始前，使用者或發起角色必須明確說明本次任務所屬的企業名稱或專案代號。

- 企業名稱或專案代號須在任務啟動時確認，不得事後補填
- 若使用者未指定，AI 員工應主動詢問，不得自行假設服務對象
- 已知企業的縮寫代號亦可接受，但須確認使用者與 AI 員工雙方對代號的認知一致

### 3.2 未指定企業 / 專案名稱時不得建立或寫入正式檔案

在企業或專案名稱未明確指定的情況下：

- 不得在 `90_正式交付成果` 建立任何資料夾或檔案
- 不得在 `91_跨員工協作區` 建立正式任務包資料夾
- 不得將分析輸出、流程知識庫、SOP 或任何交付文件標示為「已確認」
- 可進行討論、草稿規劃、範例說明，但須明確標示「尚未指定企業，為通用草稿」

### 3.3 不得將 A 企業資料寫入 B 企業資料夾

工作區若已建立多家企業的資料夾結構，任何資料操作均須確認歸屬：

- 建立、修改、搬移、複製任何檔案前，須確認目標資料夾對應的企業代號
- AI 員工在任務執行中若發現資料夾歸屬疑慮，須主動停止並向使用者確認，不得自行判斷
- 嚴禁以「通用模板複製後忘記改企業代號」為由，將資料混入其他企業資料夾

### 3.4 草稿與正式交付成果必須分流

| 類型 | 存放位置 | 說明 |
|------|----------|------|
| 草稿、協作中的中間成果、待確認的分析輸出 | `91_跨員工協作區/[企業代號]_[YYYYMMDD]_[任務名稱]/` | 未確認內容不得放入 90 |
| 已確認、可對外交付的正式版本 | `90_正式交付成果/[企業代號]/[階段或模組]/` | 僅放入已經使用者確認的版本 |
| 企業流程知識庫（員工_03 負責） | `員工_03_流程內控助理/[企業代號]/` | 知識庫版本須標示，草稿須標示 Draft |

## 4. 企業 / 專案命名規則

### 4.1 企業代號命名建議

| 規則 | 說明 | 範例 |
|------|------|------|
| 建議格式 | `[客戶代號]` 或 `[客戶代號]_[專案名稱]` | `ABC`、`ABC_ERP導入`、`XYZ_採購模組` |
| 代號字元 | 建議使用英文縮寫或數字，避免空格與特殊符號 | `ACME`、`CLIENT01` |
| 中文名稱 | 若客戶要求使用中文，可使用中文，但須全工作區統一用法 | `台灣製造_ERP`、`北部電子` |
| 不得使用 | 模糊代號如「客戶A」、「測試」、「專案」、「新公司」等，除非全工作區一致定義 | ❌ `客戶A`、❌ `新案子` |

### 4.2 任務資料夾命名格式

| 用途 | 建議格式 | 範例 |
|------|----------|------|
| 91_跨員工協作區任務包 | `[企業代號]_[YYYYMMDD]_[任務名稱]` | `ABC_20260528_採購流程分析` |
| 90_正式交付成果企業層 | `[企業代號]/` | `ABC/` |
| 90 內部階段層 | `[階段或模組名稱]/` | `01_需求分析/`、`採購模組/` |
| 員工_03 知識庫 | `[企業代號]/` | `ABC/` |

## 5. 91_跨員工協作區使用規則

### 5.1 91 的定位

`91_跨員工協作區` 是工作區的中間成果暫存與跨員工協作區，不是正式交付成果區。其核心功能包含：

- 跨員工任務交接（員工_03 → 員工_01 → 員工_02）
- 協作中的草稿存放
- 待確認事項的集中追蹤
- 任務包的標準化結構管理

### 5.2 任務包建立規則

每次新任務應在 91 建立一個任務包資料夾，命名格式為：

`[企業代號]_[YYYYMMDD]_[任務名稱]/`

任務包內部結構見第 11 章。

### 5.3 任務包管理規則

| 規則 | 說明 |
|------|------|
| 一任務一資料夾 | 同一企業不同任務不得共用資料夾，避免混淆 |
| 企業代號必填 | 資料夾名稱首碼必須包含企業代號，不得使用通用名稱 |
| 任務完成後的處理 | 已完成任務的正式成果移至 90_正式交付成果；任務包本體保留於 91 作為協作紀錄 |
| 不同企業嚴格分隔 | 同一 91 目錄下若有多個企業的任務包，資料夾命名首碼須清楚區分，不得混放 |

## 6. 90_正式交付成果使用規則

### 6.1 90 的定位

`90_正式交付成果` 是工作區唯一的正式交付品存放區。此處的所有文件均應代表「已確認、可對外交付或可向客戶提交」的版本，不得存放草稿或未確認文件。

### 6.2 分類結構規則

建議資料夾結構如下，可依專案實際階段調整子資料夾名稱，但各企業的層級邏輯應保持一致：

```
90_正式交付成果/
└─ [企業代號]/
    ├─ 01_需求分析/
    ├─ 02_流程知識庫/
    ├─ 03_SOP/
    ├─ 04_需求文件/
    ├─ 05_交付文件/
    └─ 99_其他/
```

### 6.3 存放條件

| 條件 | 說明 |
|------|------|
| 必須已有使用者確認 | 未經確認的草稿不得存入 90 |
| 版本號必須明確 | 存入 90 的文件需標示正式版本號（v1.0、v1.1 等），不得標示 Draft |
| 企業代號必須對應 | 文件所屬企業代號需與資料夾首碼一致 |
| 禁止直接複製 91 草稿 | 從 91 移入 90 前，需確認已完成確認流程，不得直接複製未確認版本 |

## 7. 三位員工資料存放原則

### 7.1 員工_03_流程內控助理

- 企業流程知識庫依企業分層，建立於：`員工_03_流程內控助理/[企業代號]/`
- 各模組文件（採購、銷售、庫存等）須置於對應企業資料夾下
- 不得將 A 企業的流程資料放入 B 企業知識庫資料夾
- 知識庫文件標題應明確標示所屬企業代號與版本

| 文件類型 | 建議存放路徑 |
|----------|-------------|
| 企業流程知識庫總覽 | `員工_03_流程內控助理/[企業代號]/00_流程知識庫總覽.md` |
| 各模組流程文件 | `員工_03_流程內控助理/[企業代號]/01_採購流程.md` |
| 待確認事項清單 | `員工_03_流程內控助理/[企業代號]/99_待確認事項清單.md` |

### 7.2 員工_01_專案分析師

- 分析輸出應存放於 91 對應任務包，或 90 對應企業資料夾（確認後）
- 每份分析輸出文件的標題頁須標示：企業代號、專案名稱、分析日期
- 若分析過程中接觸多個企業資料，需在分析文件中明確標示本文件僅適用於哪家企業
- 不得將 A 企業的需求缺口清單與 B 企業的流程知識庫混用作為分析依據

### 7.3 員工_02_文件資料助理

- 所有交付文件、SOP、需求文件、匯入模板的標題頁或文件首行須標示企業代號
- Excel / CSV 匯入模板若含有示範資料，須明確標示「[企業代號] 專用」或「範例資料，僅供示範」
- 正式版文件存放於 `90_正式交付成果/[企業代號]/` 對應子資料夾
- 草稿存放於 91 對應任務包的 `03_文件資料助理_交付文件草稿.md`

## 8. 檔案命名規則

### 8.1 正式交付文件命名格式

`[企業代號]_[文件類型]_[版本或日期].[副檔名]`

| 組成元素 | 說明 | 範例 |
|----------|------|------|
| 企業代號 | 客戶代號或專案代號，與資料夾首碼一致 | `ABC`、`XYZ` |
| 文件類型 | 清楚描述文件內容，避免「文件」、「資料」等模糊詞 | `採購SOP`、`需求缺口分析`、`匯入模板_供應商` |
| 版本或日期 | 正式版用版本號，草稿可用日期 | `v1.0`、`Draft_20260528` |

**命名範例：**

| 文件類型 | 建議檔名 |
|----------|----------|
| 採購 SOP 正式版 | `ABC_採購SOP_v1.0.md` |
| 需求缺口分析草稿 | `ABC_需求缺口分析_Draft_20260528.md` |
| 供應商匯入模板 | `ABC_Odoo匯入模板_供應商_v1.0.xlsx` |
| 採購流程知識庫草稿 | `ABC_採購流程知識庫_Draft_v0.1.md` |

### 8.2 任務包內部檔案命名

任務包內部檔案（`00_` 至 `99_`）以編號前綴為主，不需重複加企業代號（資料夾已標示）。但每份文件的標題頁或文件首行應標示企業代號，確保單獨開啟時仍可識別歸屬。

### 8.3 不可接受的命名方式

| ❌ 不可接受 | 原因說明 |
|-------------|---------|
| `需求文件.md` | 無企業代號、無版本，無法識別歸屬 |
| `草稿.xlsx` | 無任何識別資訊 |
| `SOP_v1.md` | 缺少企業代號 |
| `ABC_文件.md` | 文件類型過於模糊 |
| `copy_ABC_採購SOP_v1.0.md` | 複製檔案未清除前綴，易造成版本混淆 |

## 9. 任務啟動前確認清單

每次任務開始前，使用者或 AI 員工應確認以下所有項目：

- [ ] 本次任務的企業名稱或客戶代號已明確指定
- [ ] 企業代號已與工作區既有命名規則一致（若為新企業，已確認代號格式）
- [ ] 本次任務的任務名稱與任務日期已確認（用於建立 91 任務包資料夾）
- [ ] 是否需要在 91_跨員工協作區 建立新任務包（若是，資料夾命名已準備好）
- [ ] 本次任務的主責角色已判斷（員工_03 / 員工_01 / 員工_02 或跨員工）
- [ ] 若涉及多員工交接，交接順序已確認（員工_03 → 員工_01 → 員工_02）
- [ ] 若有既有流程知識庫可參考，已確認版本與最後更新日期
- [ ] 本次任務是否涉及多企業資料？若是，已確認每份輸入資料的來源企業

## 10. 禁止事項

| 禁止行為 | 說明 |
|----------|------|
| 禁止未指定企業名稱即建立正式檔案 | 任務啟動前必須確認企業代號，否則只能進行討論，不得建立正式資料夾或檔案 |
| 禁止將 A 企業資料寫入 B 企業資料夾 | 包含直接寫入、模板複製後修改不完整、誤用其他企業流程知識庫作為當前任務依據 |
| 禁止在未確認情況下將草稿存入 90_正式交付成果 | 90 只放已確認的版本，Draft 狀態文件一律存放於 91 |
| 禁止使用無識別性的模糊命名 | 所有正式產出均需包含企業代號，如「草稿.md」、「需求文件.xlsx」均不可接受 |
| 禁止在多企業並行時使用未分層的共用資料夾 | 每家企業必須有獨立資料夾，不得共用無識別性的通用資料夾 |
| 禁止將歷史企業的資料夾搬移至現行企業目錄 | 歷史任務資料夾須保持原有歸屬，不得因整理需求而搬移 |
| 禁止 AI 員工自行假設服務對象 | 若使用者未指定企業，AI 員工須主動詢問，不得自行判斷或預設 |

## 11. 任務包標準結構範例

以下為一個完整任務包的標準結構，供建立任務包時參考：

```
91_跨員工協作區/
└─ ABC_20260528_採購流程分析/
    ├─ 00_任務說明.md
    ├─ 01_流程內控助理_流程知識整理.md
    ├─ 02_專案分析師_需求缺口分析.md
    ├─ 03_文件資料助理_交付文件草稿.md
    └─ 99_待確認事項.md
```

**各檔案用途說明：**

| 檔案 | 主責員工 | 用途 |
|------|----------|------|
| `00_任務說明.md` | 使用者或任務發起者 | 記錄企業代號、專案名稱、任務啟動日期、任務目的、預計交付物、主責角色、知識庫參考版本 |
| `01_流程內控助理_流程知識整理.md` | 員工_03_流程內控助理 | 整理本次任務相關的流程節點、單據清單、簽核規則、內控重點、例外情境、待確認事項 |
| `02_專案分析師_需求缺口分析.md` | 員工_01_專案分析師 | 輸出需求缺口清單、P1/P2/P3 問題清單、流程斷點報告、系統影響評估、交接摘要 |
| `03_文件資料助理_交付文件草稿.md` | 員工_02_文件資料助理 | 依前兩份產出整理草稿，標示所有待確認事項，不得在前兩份未完成時自行產出 |
| `99_待確認事項.md` | 跨員工共用 | 彙整三位員工在此任務中所有「待補資料」與「需確認」項目，集中追蹤確認狀態 |

**00_任務說明.md 建議包含的欄位：**

| 欄位 | 說明 |
|------|------|
| 企業代號 | 本次任務服務的企業代號（與資料夾首碼一致） |
| 企業全名 | 企業完整名稱（僅在任務包內部使用） |
| 專案名稱 | 本次任務的專案名稱或任務描述 |
| 任務啟動日期 | YYYY-MM-DD |
| 預計完成日期 | YYYY-MM-DD 或「待定」 |
| 主責角色 | 員工_03 / 員工_01 / 員工_02 / 跨員工 |
| 任務目的 | 簡要說明本次任務的目標與觸發原因 |
| 預計交付物 | 條列本次任務預計產出的文件或資料 |
| 知識庫參考版本 | 若有員工_03 知識庫，標示版本與日期 |
| 任務包建立者 | 使用者或 AI 員工 |
| 啟動時待確認事項 | 任務啟動時已知的待確認事項 |

## 12. 後續與 README.md、CLAUDE.md、91_跨員工協作區模板的關係

本文件為「多企業 / 多專案防混淆規則」的完整規則集中文件。其他工作區文件應與本文件的規則保持一致，補強建議如下：

| 文件 | 建議補強方向 | 建議執行時機 |
|------|-------------|-------------|
| `CLAUDE.md` 第 8 章 | 任務執行流程中加入「任務啟動前必須指定企業/專案名稱」步驟，並引用本文件 | 本文件確認後執行 |
| `CLAUDE.md` 第 12 章 | 禁止事項補充本文件第 10 章的核心條款（未指定企業不得建立檔案、禁止跨企業寫入） | 本文件確認後執行 |
| `CLAUDE.md` 第 6 章 | 檔案命名規則補充客戶代號強制首碼，引用本文件第 8 章 | 本文件確認後執行 |

## 13. 總 worklog 與任務包 worklog 分層管理規則

本工作區存在兩個層級的 worklog，各有明確的記錄範圍與使用時機，不得混用。

### 13.1 兩種 worklog 的定義與分工

| 類型 | 路徑 | 記錄範圍 | 使用時機 |
|------|------|----------|----------|
| 總 worklog | `顧問Cowork/worklog.md` | 工作區層級的建置、規則、模板、Skill、人員設定與治理變更 | 任何影響整體工作區架構、規則或共用資源的操作 |
| 任務包 worklog | `91_跨員工協作區/[任務包]/worklog.md` | 單一企業 / 專案 / 任務包內部的執行過程、角色交接、內容填寫、待確認更新與交付狀態 | 任務包進入內容執行階段，或開始填寫 01 / 02 / 03 / 99 時 |

### 13.2 任務包 worklog 的建立時機

- **必須建立**：任務包開始填寫 01_流程內控助理、02_專案分析師、03_文件資料助理或 99_待確認事項任一檔案內容時
- **必須建立**：三位員工進行角色交接時
- **可暫不建立**：僅建立任務包資料夾、複製模板、填寫 00_任務說明.md 基本資訊時（此階段記錄於總 worklog 即可）

### 13.3 任務包 worklog 的放置規範

任務包 worklog.md 應放在已包含企業代號 / 專案代號的任務包資料夾內，不要求檔名本身包含企業代號。例如：`91_跨員工協作區/Kenda_20260601_採購簽核需求分析/worklog.md`，其中企業代號已由資料夾名稱承載。

### 13.4 禁止事項

- 禁止將單一企業任務包的執行細節長期寫入總 worklog.md
- 禁止任務包 worklog.md 記錄其他企業的任務資訊
- 任務包 worklog.md 的企業代號欄位必須填寫，不得留空
