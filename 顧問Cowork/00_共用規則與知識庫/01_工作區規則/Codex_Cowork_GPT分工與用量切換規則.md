## 一、核心分工

```text
GPT = 想清楚、判斷、整理、草擬指令
Cowork = 依使用者或 GPT 整理後的指令執行任務
Codex = 驗收 Cowork 寫入結果、檢查是否越權、確認紀錄同步
```

| 工具 | 定位 | 適合處理 | 不適合處理 |
|---|---|---|---|
| GPT 專案視窗 | 顧問主管 / 策略腦 | 判斷、規劃、提示詞、文件草稿、架構分析、風險審查、草擬 Cowork 指令 | 實際寫本機檔案 |
| Cowork | 執行員工 | 依使用者 / GPT / Codex 指令執行、整理文件、補 99、回報 worklog | 自己決定擴架構、自行寫知識庫、自行搬正式成果 |
| Codex | 工作區總控 / 落地驗收 | 驗收 Cowork 成果、檢查本機檔案狀態、同步或檢查 worklog、維護檔案狀態、必要時寫入交換區 | 一般聊天、長篇討論、反覆推敲 |

## 二、交換區使用原則

一般情況下，不一定由 Codex 先寫交換區 `00_Codex_to_Cowork_任務指令.md`。

允許兩種路徑：

### 路徑 A：GPT / 使用者直接給 Cowork 指令

適用情境：

- GPT 已經整理好明確指令
- 任務範圍清楚
- 不需要 Codex 先建交換區 `00`
- 目標是節省 Codex 用量

流程：

```text
GPT 整理指令
↓
使用者交給 Cowork 執行
↓
Cowork 完成並寫入結果 / worklog
↓
Codex 驗收 Cowork 實際寫入結果
```

### 路徑 B：Codex 寫交換區 00

適用情境：

- 任務需要精準落地到本機工作區
- 任務涉及多個檔案同步
- 需要明確限制 Cowork 不可做事項
- 需要 Codex 保留完整審核紀錄
- 使用者明確要求 Codex 建立交換區任務

流程：

```text
Codex 寫入 00
↓
Cowork 讀取並執行 / 或提出 01 計畫
↓
Codex 驗收 03
↓
必要時寫入 04
```

## 三、Codex 用量切換規則

| Codex 剩餘用量 | 使用策略 |
|---:|---|
| > 90% | 可正常用 Codex 做規劃 + 寫入 + 驗收 |
| 85%–90% | 開始節制，只讓 Codex 做檔案落地與驗收 |
| <= 85% | 切回 GPT 專案視窗做一般討論、判斷、策略規劃 |
| <= 70% | Codex 只處理必要寫檔、驗收、同步 worklog |
| <= 50% | 除非緊急交付，否則暫停新任務包與新 Skill 建置 |

---

## 四、必須使用 Codex 的情境

- 要驗收 Cowork 寫入結果
- 要檢查 Cowork 是否有越權改檔
- 要確認本機資料夾結構或檔案實際狀態
- 要確認 `worklog / 00 / 99` 是否正確
- 要檢查是否有非授權異動
- 要建立 / 修改 Skill 檔案
- 任務需要 Codex 寫入交換區 `00`

---

## 五、應優先使用 GPT 專案視窗的情境

- 討論下一步策略
- 判斷 Cowork 回報是否合理
- 草擬給 Cowork 的任務指令
- 拆任務優先順序
- 文件內容草擬
- Skill 架構設計的前期討論
- 顧問判斷 / 內控理解 / 風險推敲

---

## 六、節省 Codex 用量模式

當 Codex 剩餘用量 <= 85% 時，啟用節省模式：

1. 不用 Codex 做長篇討論。
2. 不用 Codex 反覆推敲任務策略。
3. GPT 先完成判斷、指令草稿與風險審查。
4. 若 GPT 指令已足夠明確，使用者可直接交給 Cowork，不必先請 Codex 寫交換區 00。
5. Cowork 執行後，Codex 只做必要驗收、越權檢查與狀態同步。
6. 只有在任務需要本機檔案落地設計、交換區紀錄或高風險限制時，才由 Codex 先寫 `00_Codex_to_Cowork_任務指令.md`。
