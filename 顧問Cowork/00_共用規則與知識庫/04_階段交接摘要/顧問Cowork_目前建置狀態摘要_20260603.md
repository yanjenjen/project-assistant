## 五、目前可正式使用的流程

1. **任務包建立流程：** 啟動前檢查表 → 00 任務說明 → 01/02/03/99 填寫 → worklog 紀錄 → 跨員工交接
2. **資料進場流程：** 確認企業代號 → 敏感等級判斷 → file-naming-and-classification-advisor 產出建議表 → 使用者確認 → 改名搬移 → 00 資料進場紀錄補記
3. **範本萃取流程：** document-template-pattern-extractor → 六大面向萃取 → 草稿輸出暫存區 → 使用者確認 → 可沉澱時寫入 02_共用模板
4. **規格書撰寫流程：** 依規格書撰寫規則 v0.1 → 判斷文件類型（標準作業 vs API）→ 填寫五大章節 → 標示待確認事項 → 整理 A/B/C 三層清單 → A 層決策確認表交客戶
5. **檔案命名與歸檔流程：** 建議表 → 使用者確認 → 第一批改名 → 第二批改名 → 第三批改名+搬移；完整執行經驗已驗證

## 八、下一個真實任務包啟動建議

**可以啟動真實任務包的判斷標準（目前全部具備）：**

- ✅ 任務串接模板完整（00/99/worklog + 啟動前檢查表）
- ✅ 多企業防混淆規則已建立
- ✅ 資料進場流程已標準化
- ✅ 規格書撰寫規則已有 v0.1 草稿可參照
- ✅ file-naming-and-classification-advisor 可協助資料整理
- ✅ worklog 分層管理已到位

**啟動第一個真實任務包的建議步驟：**

1. 確認企業代號與任務名稱
2. 完成啟動前檢查表（敏感等級判斷、任務範圍確認）
3. 依企業代號在 `01_原始資料_不直接處理/` 建立子資料夾
4. 資料進場後，使用 `file-naming-and-classification-advisor` 產出建議表並確認
5. 建立任務包於 `91_跨員工協作區/{企業代號}_{日期}_{任務名稱}/`
6. 依三角協作流程（海瑟姆 → 砂砂 → 小果）執行任務
7. 若有規格書需求，依規格書撰寫規則 v0.1 撰寫並整理待確認清單
8. 任務完成後，啟動知識沉澱判斷，將可去識別化的通用知識寫入 `02_通用流程知識庫/`

**特別提醒：** 第一個真實任務包完成後，可同步評估 `file-naming-and-classification-advisor` 是否達到升版條件（真實多專案資料進場測試）。
