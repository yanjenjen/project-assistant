# 02_通用流程知識庫｜使用說明

---

## 1. 本區用途

本資料夾用於收錄不包含特定企業識別資訊的通用 ERP 流程知識，供三位 AI 員工在不同企業任務包中引用與參考。

目標是讓過去任務包中沉澱的流程觀察、常見缺口與內控重點，在下一個任務包中能被快速取用，縮短流程知識整理的時間，提升分析品質一致性。

---

## 2. 可收錄內容

以下內容在去除企業識別資訊後，可收錄至本知識庫：

- 通用 ERP 業務流程結構（如採購循環、銷售循環、庫存流程、應付應收循環）的節點描述
- 各流程節點的常見輸入單據、輸出單據與負責角色（以角色類型描述，不使用企業內部職稱）
- 常見簽核與內控控制點（如採購金額授權分級、驗收三向比對）
- 各流程常見缺口類型（如「金額授權層級未分級」、「驗收後財務通知機制不明確」）
- 常見例外情境與處理方式（如緊急採購補單）
- Odoo 模組與通用流程節點的對應關係（不包含客製化設定）
- 常見風險點與建議對策（以通用描述，不涉及企業具體數字）

---

## 3. 不可收錄內容

以下內容屬企業專屬，禁止放入本知識庫：

- 企業名稱、品牌名稱、組織架構、人員姓名或職稱
- 企業內部的具體金額數字、授權金額級距、付款條件
- 企業客製化的單據名稱、流程步驟或系統設定參數
- 任何可反推出服務對象身份的業務特徵描述
- 客戶提供的原始文件、會議紀錄或訪談內容
- 90_正式交付成果 中的任何文件內容
- 任務包 99_待確認事項.md 的具體企業待確認項目

**判斷標準**：若提取後仍可辨識「這是哪家公司的規則或流程」，則不可收錄。

---

## 4. 企業資料脫敏與抽象化規則

從任務包中提取知識時，須依以下規則進行抽象化處理：

| 原始內容類型 | 抽象化方式 |
|------------|-----------|
| 企業名稱 | 以「企業」或「客戶」代替，或以行業類型描述（如「製造業客戶」） |
| 具體金額數字 | 改為「金額 X」或「高 / 中 / 低金額」等相對描述 |
| 內部職稱 | 改為通用角色類型（如「採購人員」、「財務主管」） |
| 特定系統模組名稱 | 僅保留通用模組類型（如「採購模組」而非客製功能名稱） |
| 具體業務規則 | 改為「某企業採用的規則之一」並說明適用情境 |

---

## 5. 知識沉澱流程

1. **任務包完成或重要階段結束後**，對應員工初步評估本次是否有可沉澱的通用知識
2. **在對話中提出知識摘要草稿**，標示來源任務包（企業代號）與脫敏方式
3. **使用者確認後**，由對應員工建立或更新知識庫檔案
4. **同步更新總 worklog**，記錄知識庫新增或修改的摘要
5. 知識庫檔案應標示版本、建立日期、來源類型（如「通用流程觀察」）

---

## 6. 由誰負責初步判斷

| 知識類型 | 初步判斷角色 |
|----------|-------------|
| 流程結構、內控點、例外情境 | 員工_03_流程內控助理（海瑟姆） |
| 需求分析框架、常見缺口分類 | 員工_01_專案分析師（砂砂） |
| 通用文件框架、SOP 章節結構 | 員工_02_文件資料助理（小果） |

---

## 7. 寫入前必須經使用者確認

本知識庫所有檔案的建立與修改，均需先在對話中呈現草稿，等待使用者確認後方可寫入。

禁止在使用者未確認的情況下，自行將任務包內容提取至本知識庫。

**特別禁止事項：測試任務包示範內容不得收錄**

- 測試任務包（資料夾命名含 `Test_*` 或明確標示為測試任務者）中的示範流程、示範需求、示範缺口、示範 SOP，不得直接收錄為通用流程知識。
- 原因：測試任務包內容主要用於驗證工作流程與模板結構，不代表真實企業觀察、真實專案經驗，也未經實際專案驗證，無法作為通用顧問知識的依據。
- 若日後需引用測試任務包中的某個概念或流程架構，必須重新以真實專案、正式客戶資料或使用者確認後的通用規則為依據，重新撰寫後方可收錄。
