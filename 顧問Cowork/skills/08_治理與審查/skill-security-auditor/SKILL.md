# skill-security-auditor｜Skill 安全審查

> 版本：v0.1.0｜狀態：試行中
> 主要用途：Skill 本身的安全性靜態審查，提供風險提示與修正建議。不等同正式治理決策。

---

## 一、Skill 定位

專門審查 Skill 本身的安全性設計，職責如下：

- 檢查觸發條件是否過寬、寫檔權限是否明確、是否可能自動介入任務、是否與其他 Skill 衝突、是否碰高風險檔案、是否可能造成敏感資料誤寫、是否有清楚回復 / 封存方式
- **僅在使用者明確要求 Skill 安全審查時啟動**，不得自動介入任何 Skill 安裝或執行流程
- **不得作為 Skill 安裝守門員自動攔截流程**
- **本 Skill 僅進行靜態審查與風險提示**，不得自行修改、封存、停用或刪除任何 Skill；所有修正動作皆需使用者明確授權

---

## 二、使用情境

啟用本 Skill 的情境：

- 安裝新 Skill 前，審查 SKILL.md 的安全性設計
- 既有 Skill 修改後，確認改動是否引入新風險
- 定期健檢工作區內所有 Skill 的安全性設計
- 懷疑某 Skill 觸發條件過寬、會亂寫檔、或與其他 Skill 有衝突時，主動呼叫審查

---

## 三、觸發條件

**主要觸發詞**（推薦使用，意圖最明確）：
- `skill-audit：`
- `審查這個 Skill：`
- `檢查這個 Skill 安不安全：`

**輔助觸發詞**（語境明確為 Skill 安全審查時啟用）：
- `這個 Skill 的觸發條件有沒有問題`
- `這個 Skill 會不會亂寫檔案`

**觸發限制：**
- **僅在使用者明確要求 Skill 安全審查時啟動**
- 不得因 Skill 安裝、修改、執行而自動介入
- 不得在使用者未主動呼叫的情況下主動建議「要不要安全審查一下」並自行執行

---

## 四、不適用情境

以下情境**不應使用本 Skill**：

- **評估一般顧問文件品質**：請使用 `consultant-output-evaluator`
- **任務規劃**：請使用 `erp-task-planning-assistant`
- **建立、修改、封存、停用 Skill**：本 Skill 不負責執行，需另行授權
- **自動攔截 Skill 安裝流程**：本 Skill 不得介入安裝流程
- **取代使用者的最終治理決策**：本 Skill 提供建議，不做最終決定

---

## 五、可審查對象

| 對象 | 審查重點 |
|---|---|
| 新安裝前的 SKILL.md | 觸發條件、寫檔行為、回復方式是否設計完整 |
| 既有已安裝 Skill | 觸發詞是否已過時或過寬、寫檔限制是否仍有效 |
| 修改後的 Skill | 改動是否引入新的觸發風險或寫檔風險 |
| 跨 Skill 觸發詞 / 職責衝突 | 多個 Skill 的觸發詞是否重疊，職責是否相互干擾 |

---

## 六、審查維度

| 維度 | 定義 | 不通過的典型情境 |
|---|---|---|
| 寫檔權限 | SKILL.md 是否明確聲明預設不寫檔；有條件寫檔時條件是否清楚 | 未聲明預設不寫檔；或「若使用者要求」條件模糊 |
| 觸發條件是否過寬 | 觸發詞是否可能在非預期情境下啟動 | 觸發詞為單詞（如「分析」、「整理」），容易誤觸 |
| 是否可能自動介入任務 | 是否明確聲明僅在明確呼叫時啟動 | 未聲明觸發限制；觸發設計可能在一般對話中意外啟動 |
| 是否與其他 Skill 衝突 | 觸發詞或職責範圍是否與既有 Skill 重疊 | 兩個 Skill 能被相同語句觸發 |
| 是否修改高風險檔案 | 是否聲明不碰 CLAUDE.md / AGENTS.md / 90_正式交付成果等 | 未明確排除高風險檔案；或有條件碰但條件不清楚 |
| 是否可能造成敏感資料誤寫 | 是否有將客戶資料、帳密、合約等寫入不必要位置的風險 | Skill 輸出含識別資訊卻預設寫入 worklog 或共用區 |
| 是否有清楚回復 / 封存方式 | SKILL.md 是否有完整的回復步驟 | 無回復章節；或步驟不完整（缺少更新 README / 導入計畫） |

## 七、與 `consultant-output-evaluator` 的邊界

| 面向 | consultant-output-evaluator | skill-security-auditor |
|---|---|---|
| 評估對象 | 顧問工作成果（文件、分析、任務計畫、Skill 輸出） | Skill 本身的安全性設計（SKILL.md） |
| 評估視角 | 顧問交付品質（完整性、正確性、可追蹤性） | 系統安全性（觸發、寫檔、衝突、回復） |
| 可否串接 | ✅ 可串接：先用 auditor 審查 Skill 安全性，再用 evaluator 評估 Skill 輸出品質 | — |
| 是否互相取代 | ❌ 不得互相取代，各有職責邊界 | — |

## 九、寫檔限制

- **預設只在對話中輸出**，不自動寫入任何檔案
- 不自動寫入 README、導入計畫、worklog、任務包或正式文件
- 若使用者明確要求寫入，需另行確認**寫入位置與範圍**後才執行

---

## 十、回復方式

若安裝後觸發條件出現錯誤、被誤用為自動守門員，或與其他 Skill 觸發混淆：

1. 立即停止使用本 Skill
2. 向使用者回報問題描述（誤觸情境、混淆的 Skill 名稱）
3. 提出停用 / 封存計畫，等待使用者授權
4. 授權後將 `skills/08_治理與審查/skill-security-auditor/` 移至 `skills/99_封存/`
5. 若 `08_治理與審查/` 內已無其他 Skill，一併評估是否移除空資料夾（需另行授權）
6. 同步更新 `skills/README.md` 索引與 `skills/第一階段Skill導入計畫_v0.1.md` 狀態
