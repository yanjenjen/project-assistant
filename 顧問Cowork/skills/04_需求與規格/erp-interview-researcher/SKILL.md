# erp-interview-researcher｜ERP 顧問訪談前準備助理

> 版本：v0.1.0
> 狀態：🔶 試行中（尚未完成實際任務壓測）
> 主責角色：員工_01_專案分析師（訪談準備輔助工具）
> 建立日期：2026-06-12

---

## 一、Skill 定位

`erp-interview-researcher` 是 ERP 顧問在**正式訪談前**的「訪談準備與問題設計輔助工具」。

**核心功能：** 給我訪談對象角色 + 模組或流程範圍，我產出結構化問題清單草稿，讓顧問帶著清楚的提問目的進入訪談。

**研究對象：** 客戶企業的流程角色（採購主管、業務人員、倉管、財務、IT 管理員等）
≠ Jenny 本人（由外層 `user-research` 負責）
≠ ERP 系統本身（由 `odoo-ui-analyzer` 負責）

**架構位置（上下游串接）：**

```
odoo-ui-analyzer（截圖脈絡，可選輸入）
        ↓
erp-interview-researcher（訪談前問題設計）
        ↓ 顧問執行訪談
erp-meeting-record-writer（輔導備忘錄）
        ↓
員工_03 流程內控助理（知識庫整理）
```

---

## 二、解決的 ERP 顧問工作問題

- 每次訪談問題全靠個人經驗，無標準化起點，易遺漏關鍵節點（簽核條件、例外情境、代理規則）
- 不同模組（採購 / 銷售 / 庫存 / 生產 / 會計）的訪談問題結構各異，缺乏可複用框架
- 有 `odoo-ui-analyzer` 截圖分析結果後，不知道如何轉換成訪談確認問題
- `erp-task-planning-assistant` 的「需求訪談與分析」任務包只規劃任務，不產出實際問題清單

---

## 三、適用情境

| 情境 | 說明 |
|---|---|
| AS-IS 流程訪談前準備 | 需要訪談客戶某部門主管或流程負責人，了解現行流程 |
| 特定模組需求確認訪談 | 確認採購、銷售、庫存、生產、會計、人資等單一模組的現行規則 |
| odoo-ui-analyzer 截圖延伸確認 | 截圖分析發現疑問點，需設計訪談問題向客戶確認 |
| GAP 分析前的流程確認 | 在進行缺口分析前，先透過訪談補齊流程知識 |
| 簽核規則 / 例外情境專項訪談 | 現行流程大致已知，但簽核層級與例外處理尚未確認 |

## 四、不適用情境

| 情境 | 應使用工具 |

| 訪談中速記整理、訪談後答案整理（含當場筆記） | 視情境：員工_03 流程內控助理（知識整理）或 `erp-meeting-record-writer`（備忘錄） |

| Jenny 本人工作習慣研究 | `user-research` |

| 正式需求規格書或缺口分析報告 | 員工_01_專案分析師 |

> **【E-001 說明】** 本 Skill 只處理訪談「前」。訪談中或訪談後產生的任何答案整理、速記處理、知識庫更新，均不由本 Skill 負責，應移交員工_03 或 `erp-meeting-record-writer`。

### 主觸發詞（精確觸發）

- `erp-interview：`

### 輔觸發詞（需雙重信號）

以下觸發詞需搭配 **ERP 模組名稱 / 流程名稱 / 客戶角色** 才啟用：

- `幫我設計 ERP 客戶訪談問題`

### 觸發邊界防護

> **【E-002 說明】**

- **「幫我設計訪談問題」** 單獨出現**不觸發**（過寬，可能是 HR 面試、使用者研究、學術訪談）

- `user-research` 觸發詞為 `了解我的習慣：`、`user-research：`，研究對象為 Jenny 本人，與本 Skill 觸發詞不重疊

## 六、輸入資料要求

### 6.1 必要輸入（缺少則先詢問，確認後才進入產出流程）

| 欄位 | 說明 |
|---|---|
| 企業代號或客戶名稱 | **必須優先確認，確認後才進入產出流程**。未提供時必須先詢問使用者確認；使用者明確表示暫不提供，才可標示 `[待確認：企業代號]` 並繼續，不得靜默省略。 |
| 訪談對象角色 | 例：採購主管、業務助理、倉管人員、財務主管、IT 管理員 |
| 訪談範圍 | 模組或流程範圍，例：採購流程、銷售訂單至出貨、庫存盤點 |

### 6.2 選填輸入（有益但非必要）

| 欄位 | 說明 |
|---|---|
| `odoo-ui-analyzer` 截圖分析輸出 | 截圖分析觀察點可轉換為系統確認問題（見第七節 E-003） |
| 已知疑問或待確認清單 | 可從員工_03 知識庫「待補資料」或「需確認」項目取用 |
| 訪談時間限制 | 影響問題數量規劃（30 分鐘 vs 2 小時） |
| 已知的企業現行做法摘要 | 避免問重複問題 |

### 6.3 禁止

- 不得補造企業未提供的流程假設
- 不得引用其他企業的知識庫內容
- 每次只產出一個企業的訪談準備清單（不得混用）

## 七、預設輸出格式

### 7.1 格式模板

```
# [企業代號] 訪談準備清單
訪談對象角色：{角色}
訪談範圍：{模組/流程}
準備日期：{YYYY-MM-DD}
狀態：草稿，待顧問確認
版本：v0.1 Draft

## 開場確認問題（2–3 題）
確認角色職責、流程參與範圍、資料來源

## 核心流程問題（5–10 題，依模組範圍）
流程起點、觸發條件、步驟、輸出單據

## 簽核與權責確認問題（3–5 題）
簽核層級、授權條件、代理規則、越級情境

## 例外情境問題（2–4 題）
已知例外、緊急處理、特殊客戶/供應商規則

## 系統操作確認問題（條件產出，見下方說明）
截圖觀察點轉換的確認問題

## 本次訪談目標確認項目
列出本次訪談預計澄清的問題面向，供顧問對焦訪談目的，不作為知識庫更新依據
```

### 7.2 格式規則

- 每個問題以編號呈現（Q1、Q2…），每題附問題意圖說明（一行）
- 問題使用**開放式問法**，不預設客戶答案（見 R-007）
- 無裝飾性分隔線
- 雙軌 `[待確認]` 標示：正文中標記 + 文末彙整清單
- 問題數量依訪談時間限制動態調整；無時間限制時依模組規模給出建議數量
- 問題設計可參考 ERP 標準模組流程節點作為起點，但標準流程節點只作為**提問框架**，不得作為客戶現況的結論或假設；禁止在問題中使用「你們應該有…」或「根據標準流程，你們…」等預設語句。

### 7.3 系統操作確認問題的條件產出

> **【E-003 說明】**

- **只有使用者提供 `odoo-ui-analyzer` 截圖分析輸出或明確 Odoo 畫面觀察點時**，才產出「系統操作確認問題」區塊
- 未提供截圖分析時，此區塊自動省略，不得留空殼或補造假設性系統問題
- 問題數量上限：0–3 題（依截圖觀察點數量決定）

### 7.4 本次訪談目標確認項目說明

> **【E-004 說明】**

- 此區塊列出「本次訪談預計澄清的問題面向」，供顧問對焦訪談目的
- 此區塊**不等於**知識庫缺口分析，不作為員工_03 知識庫的更新依據
- 本 Skill **不整理訪談答案**、**不寫入流程知識庫**、**不產出待補資料清單的更新版本**

## 九、寫檔限制

> **【E-005 說明】**

- 訪談準備清單屬**草稿輔助工具**，**預設不自動寫檔**，在對話中輸出後由使用者決定是否儲存
- 使用者授權寫檔時，檔名格式：`{企業代號}_訪談準備清單_{角色或模組}_{YYYYMMDD}_Draft.md`
  - 檔名**必須包含企業代號或專案代號作為首碼**，不得省略
- 寫入位置：`05_暫存區/` 或任務包草稿區
- **禁止寫入以下位置：**
  - `90_正式交付成果/`（草稿輔助工具不屬正式交付範疇）
  - `03_公司範例模板/`（原始範本區禁止 AI 寫入）
  - `員工_03_流程內控助理/` 下的知識庫（知識庫由員工_03 負責）
- 企業代號未確認前，禁止產出任何帶企業名稱的輸出或寫檔

## 十、風險點

| 風險代號 | 風險描述 | 等級 | 緩解方式 |
|---|---|---|---|
| R-001 | 問題清單草稿被誤當作確認後的流程知識，直接移入知識庫 | 高 | 輸出必須標示草稿狀態；第七節 7.4 明確宣告不整理訪談答案、不寫入知識庫 |
| R-002 | 跨企業污染：訪談問題引用了其他企業的流程假設或知識庫 | 高 | 企業代號確認後才進入產出；問題清單必須包含企業代號首碼；禁止引用其他企業資料 |
| R-003 | 補造問題：根據「一般企業通常…」推測出不適用的問題，導致訪談時間浪費或誤導客戶 | 中 | 問題來源只能是：訪談對象角色特性、模組標準流程節點（作為框架而非結論）、使用者提供的已知疑問、odoo-ui-analyzer 觀察點 |
| R-004 | 與 user-research 混淆，誤用訪談問題設計功能來研究 Jenny | 中 | 觸發詞設計包含 ERP 脈絡雙重信號；第一節明確宣告研究對象為客戶企業角色 |
| R-005 | 越界整理：訪談結束後被要求直接從問題清單整理出知識庫內容，取代員工_03 | 中 | 第四節不適用情境明確封閉；訪談後整理應啟用員工_03 |
| R-006 | 觸發詞「訪談準備」過於寬泛，誤觸非 ERP 情境 | 低 | 輔觸發詞需搭配模組名稱或角色名稱才啟用；「訪談準備」單獨出現不觸發（見第五節） |
| R-007 | 以標準模組流程節點作為訪談框架，被誤讀為對客戶現況的假設描述 | 中 | 標準流程節點只作為**提問框架的起點**，不作為客戶現況結論；所有問題使用開放式問法，不預設客戶答案；禁止在問題中描述「你們應該有…」或「根據標準流程，你們…」 |

## 十一、回復方式 / 停用方式

### 停用條件

以下任一成立時，應考慮停用本 Skill：

- 本 Skill 多次產出越界內容（例如：整理訪談答案、寫入知識庫）
- 觸發詞持續誤觸非 ERP 訪談情境
- 已有新版 Skill 替代

### 停用流程

1. 使用者確認停用
2. 將 `erp-interview-researcher/` 資料夾移至 `skills/99_封存/`，資料夾名稱加封存日期
3. 更新 `skills/README.md` 索引表，狀態改為「封存」並記錄日期

### 問題回報

若輸出不符合預期（問題清單格式錯誤、越界整理答案、誤觸觸發詞），請記錄：

- 觸發詞或使用者輸入原文
- 實際輸出問題描述
- 交由負責角色（員工_01）判斷是否需要修正 SKILL.md
