# erp-task-planning-assistant｜ERP 顧問任務規劃助理

> 版本：v0.1.0｜狀態：試行中
> 主要用途：任何 ERP 顧問任務開始前的標準規劃節奏，確保範圍清楚、前提確認、
> 風險可見，再決定是否進入執行階段。

> **分類說明**：本 Skill 存放於 `04_需求與規格` 分類下，但實際用途為
> **ERP 顧問任務開始前的規劃入口**，不等同於規格書撰寫 Skill。
> 規格書撰寫屬於後續執行階段，應依任務類型選用對應的執行 Skill。

---

## 使用時機

當使用者遇到以下任一情境時，啟用此 Skill：

- 收到新的客戶需求或內部任務，尚未確認範圍與執行步驟
- 任務看起來簡單，但不確定是否涉及多個流程或多位角色
- 需要確認「這個任務應該由哪位 AI 員工主責」
- 需要在建立任務包前先整理前置條件
- 不確定這個任務是否需要參考現有流程知識庫
- 初步拆解任務、學習概念、對話討論規劃方向

## 快捷啟動語

不需要輸入完整說明，使用以下短句即可啟動本 Skill：

| 快捷語 | 用途 | 企業代號狀態 |
|---|---|---|
| `任務規劃：{任務描述}` | 快速啟動，企業代號由 Step 1 詢問 | 未指定，Step 1 會詢問 |
| `先幫我規劃這件事：{任務描述}` | 口語觸發，同上 | 未指定，Step 1 會詢問 |
| `任務規劃｜企業：{代號}｜內容：{描述}` | 一次提供企業代號與任務描述，直接進入 Step 2 | ✅ 已提供 |
| `任務規劃｜企業未確認｜內容：{描述}` | 明確宣告企業代號未確認，產出暫定規劃（情況 A） | ⚠️ 明確未確認 |

**快捷語解析規則：**

- 偵測到 `任務規劃：` 或 `先幫我規劃這件事：` → 啟動本 Skill，從 Step 1 開始執行
- 偵測到 `任務規劃｜企業：X｜內容：Y` → 企業代號設為 X，任務描述設為 Y，跳過 Step 1 詢問，直接進入 Step 2
- 偵測到 `任務規劃｜企業未確認｜內容：Y` → 預設視為情況 A，產出暫定規劃，不詢問企業代號

  > ⚠️ **情況 B 安全限制**：若使用者輸入 `任務規劃｜企業未確認｜內容：___`，但內容中同時出現「建立任務包、寫入檔案、放入資料夾、修改文件」或任何正式檔案操作要求，**不得視為情況 A**。應改判斷為**情況 B**，停止一切操作，並回報：
  > 「本次任務涉及正式執行，企業代號或專案名稱為必要前提。請先確認後再繼續規劃，以避免資料誤入錯誤企業資料夾。」

快捷語不影響 Skill 的禁止事項與安全規則，所有規劃仍須遵守 `CLAUDE.md` 與本 Skill 的執行限制。

## 不適用情境

- 使用者已明確知道要做什麼，只需要直接執行（使用對應 Skill 或直接操作）
- 任務屬於單一、明確的文件格式操作（例如「幫我改這個表格欄位」）
- 屬於趕工情境、時間不足、需要立即收斂（改用 `project-fast-delivery-assistant`）

## 與其他 Skill 的邊界

| Skill | 定位 | 差異 |
|---|---|---|
| `erp-task-planning-assistant`（本 Skill） | 任務開始前的**標準規劃節奏** | 完整確認範圍、依賴、風險，產出規劃書，判斷是否建任務包 |
| `project-fast-delivery-assistant` | **趕工情境的快速收斂** | 時間有限時直接拆 P0/P1/P2，不強求完整規劃流程 |

兩者不互相呼叫，不自動串接，由使用者依情境選擇。

## 標準工作流程

收到規劃需求後，**依序執行，不跳步驟**。

### Step 1：前置確認

| 確認項目 | 確認方式 | 若未確認 |
|---|---|---|
| 企業代號或專案名稱 | 詢問使用者，不自行假設 | 見下方「企業代號未確認的處理方式」 |
| 任務描述是否足夠明確 | 判斷是否能識別任務類型與範圍 | 請使用者補充，標示「待補資料」 |

#### 企業代號未確認的處理方式

若企業代號或專案名稱尚未確認，依任務性質分兩種情況處理：

**情況 A：任務為初步拆解、學習、概念判斷或對話規劃**

可繼續產出「**暫定任務規劃**」，但必須：
- 在輸出頂端標示：`⚠️ 企業代號未確認｜規劃狀態：暫定`
- 所有步驟均以「假設情境」或「通用流程」說明，不帶入特定企業資料
- 明確告知：正式執行前必須補上企業代號或專案名稱
- **不建立任務包**
- **不進行任何檔案操作**
- **不寫入企業專屬資料夾**

**情況 B：任務涉及正式執行、需要寫入檔案或建立任務包**

停在此步驟，回報：
> 「本次任務涉及正式執行，企業代號或專案名稱為必要前提。
> 請先確認後再繼續規劃，以避免資料誤入錯誤企業資料夾。」

### Step 2：判斷任務類型

確認本次任務屬於下列哪一類（可複選）：

| 類型 | 說明 | 建議主責 |
|---|---|---|
| 流程知識建置 | 建立或更新企業流程知識庫 | 員工_03 |
| 需求訪談與分析 | 客戶需求整理、GAP 分析、功能範圍確認 | 員工_01 |
| 規格書撰寫 | 技術或業務規格書草稿 | 員工_01 / 員工_02 |
| 文件交付 | SOP、會議紀錄、匯入模板、正式交付品 | 員工_02 |
| 知識庫補強 | 補充現有流程知識庫的例外、風險、系統對應 | 員工_03 |
| 跨角色協作任務 | 涉及兩位以上 AI 員工，需要交接規劃 | 視任務決定 |
| 初步概念討論 | 尚在探索階段，不涉及正式執行 | 視情況 |

### Step 3：識別影響範圍

依任務類型，識別本次任務可能涉及的範圍：

- **流程**：哪些業務流程受影響（採購 / 銷售 / 生產 / 財務 / 庫存 / 其他）
- **文件**：需要新增、修改或參考哪些文件
- **知識庫**：是否需要先讀取或更新企業流程知識庫
- **角色**：哪位 AI 員工主責，是否需要跨角色交接
- **資料夾**：預計操作哪些路徑（91_跨員工協作區 或 90_正式交付成果）

若資料不足以判斷，標示「待補資料」，不自行推測。

### Step 4：判斷依賴關係與前置條件

列出「必須先完成才能進行本任務」的項目：

- 企業流程知識庫是否已建立（若需引用）
- 客戶確認是否已取得（若涉及需求）
- 前一階段任務包是否已交付（若為接續任務）
- 相關原始資料是否已進入工作區（若需處理資料）

### Step 5：識別風險與待確認事項

必須明確列出以下類型的項目：

- 資料不足或尚待補充的項目 → 標示「待補資料」
- 需要使用者確認才能繼續的判斷 → 標示「需確認」
- 可能影響多企業資料隔離的風險 → 明確警示
- 任務範圍模糊、可能無限擴大的風險 → 建議收斂邊界

### Step 6：產出任務規劃書

輸出格式見下方「輸出格式」章節。

### Step 7：判斷是否建議建立任務包

依據以下標準，輸出明確建議：

| 條件 | 建議 |
|---|---|
| 企業代號已確認 + 任務涉及多步驟 + 需要交接紀錄 | ✅ 建議建立任務包 |
| 任務單一、可在對話中完成、無需歸檔 | ⬜ 不需建立任務包 |
| 企業代號未確認、任務為初步概念討論 | ⚠️ 暫定規劃，正式執行前須補企業代號 |
| 企業代號未確認、任務涉及正式執行 | 🚫 不得建立任務包 |

**本 Skill 只給出建議，不自行建立任務包。建立任務包須由使用者確認後另行執行。**

- 任務描述明確度：✅ 足夠 / ⚠️ 需補充（附說明）

## 任務類型
（依 Step 2 判斷，可複選）

## 影響範圍
- 流程：
- 文件：
- 知識庫：
- 涉及角色：
- 預計資料夾路徑：

## 依賴關係與前置條件
（逐項列出，已完成 / 待確認 / 待補資料）

- 需確認：

## 禁止事項

- **不執行任何檔案操作**（不新增、不修改、不搬移、不刪除任何檔案）
- **不在企業代號未確認且任務涉及正式執行時繼續規劃**
- **不自動呼叫其他 Skill**（包含 `project-fast-delivery-assistant`、`odoo-large-feature-dev` 等）
- **不自動啟動員工_01 → 員工_02 → 員工_03 → 99 流程**
- **不自行建立任務包**，只給出「是否建議建立」的判斷
- **不推測企業代號**，必須由使用者提供
- **不將待補資料自行補齊**，一律標示「待補資料」或「需確認」
- **不擴大任務範圍**，僅規劃使用者明確提出的任務
- **不取代使用者的最終決策**
- **不將暫定規劃寫入企業專屬資料夾**
