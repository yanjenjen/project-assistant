# odoo-ui-analyzer｜Odoo 畫面顧問分析

> 版本：v0.1.0｜狀態：試行中
> 主要用途：Odoo 畫面截圖 / 描述的顧問式辨識與分析，不涉及程式開發。

---

## 一、Skill 定位

協助 ERP 顧問分析 Odoo 畫面，辨識：
- **畫面類型**（列表 / 表單 / 看板 / 儀表板 / 彈窗）
- **欄位用途**（欄位名稱、類型推測、業務意義）
- **操作按鈕與動作**（可能觸發的狀態流、動作類型）
- **控制點**（權責控制、單據控制、系統控制、例外控制）
- **ERP 需求確認點**（訪談問題、後續規格書輸入）

> 雖放在 `01_技術開發` 分類（與 Odoo 相關 Skill 同區），但定位是「Odoo 畫面顧問分析」，**不是程式開發 Skill**，不產出程式碼。

---

## 二、使用情境

啟用本 Skill 的情境：

- 顧問取得 Odoo 畫面截圖，需快速辨識欄位名稱與用途
- 需求訪談前，想了解客戶使用的模組畫面結構，準備訪談問題
- 導入規劃時，需確認某模組是否有對應的 ERP 控制點
- 整理規格書前，需先彙整畫面欄位清單作為輸入資料
- 判斷哪些畫面資訊需要客戶 / 工程師 / 原始碼進一步確認

**主要觸發詞**（推薦使用，意圖最明確）：
- `odoo-analyze：`
- `分析這個 Odoo 畫面`
- `幫我看這個 Odoo 畫面`

**輔助觸發詞**（語境明確為畫面分析時啟用）：
- `這個截圖的欄位`
- `這個模組有哪些欄位`

- `Odoo 畫面分析`

> 設計原則：觸發詞明確帶有「Odoo」或「畫面 / 截圖」語意，避免在使用者詢問一般 UI 問題或程式開發問題時意外啟動。

## 五、不適用情境

以下情境**不應使用本 Skill**：

- **直接撰寫規格書**：分析完成後若需產出規格書，請接續使用 `odoo-module-spec-writer`
- **大型功能開發規劃**：請使用 `odoo-large-feature-dev`
- **任務規劃**：請使用 `erp-task-planning-assistant`
- **非 Odoo 系統畫面**：本 Skill 僅針對 Odoo 模組畫面，不適用其他 ERP 系統截圖
- **以畫面推測當成後端邏輯定論**：本 Skill 的分析基於畫面資訊，後端流程、按鈕邏輯、權限等需另行確認，不得直接引用為規格定論

## 七、判斷可信度規則

| 可信度 | 說明 |
|---|---|
| 高 | 畫面資訊清楚，可直接判斷（例：欄位名稱與用途一致） |
| 中 | 依 Odoo 常見邏輯推測（例：`state` 欄位通常控制單據狀態流） |
| 低 | 需查看設定、原始碼或客戶實際操作流程才能確認 |

> 低可信度項目**必須**標示於「待確認事項」區塊，不得直接作為規格依據。

## 八、Odoo 畫面判讀保守規則

以下 4 類情境容易產生過度推測，分析時必須遵守對應限制。

### 8-1 App 首頁可見性規則

- Odoo 首頁只顯示使用者**釘選**的 App 圖示，不等於系統已安裝或未安裝的完整清單。
- **禁止**：「畫面未出現採購模組 → 未安裝採購模組」。
- **應改寫為**：「採購模組未出現在目前首頁選單，是否已安裝但未釘選、確實未安裝、或無獨立 App 入口，需進入 Apps / 技術設定 / 權限設定確認。」
- 模組可能以相依形式安裝（無獨立圖示）；也可能依用戶權限決定是否可見。

### 8-2 客製欄位判斷規則

- 客製模組（model name 非標準 `sale.order` / `purchase.order` 等）的欄位名稱，**不得直接套用標準模型名稱**。
- **禁止**：客製模組的「公司別」直接寫死為 `res.company`。
- **應改寫為**：「可能為 `res.company` 或客製主檔，需查看 model 定義確認。」
- 此規則同樣適用於任何與標準模型名稱「語意接近但尚未確認」的欄位。

### 8-3 Smart Button / 統計標示規則

- 畫面上的數字標示、金額符號、smart button 計數，**不得直接推定單據類型**。
- **禁止**：「$2 → 代表 2 筆銷售或財務紀錄。」
- **應改寫為**：「可能為某類關聯紀錄數（報價單 / 訂單 / 發票 / 服務票等），具體來源需進入表單查看 smart button 對應模型確認。」（可信度：低）
- 此類標示在不同模組、不同視圖中的來源差異大，一律列入「待確認事項」。

### 8-4 顧問語氣規則

- ERP 需求建議應盡量轉成**訪談問題**，而非直接下結論或用教學語氣陳述。
- **避免**：「建議補齊產品分類、料號、單位欄位，以支撐庫存與會計流程。」
- **應改寫為**：「客戶是否有產品分類、料號、UoM 的管理需求？需於訪談中確認。」
- 優先使用下列表述：「需確認」、「建議訪談確認」、「若流程如此設計」、「待客戶說明後方可判斷」。
- 不得對尚未訪談確認的流程邏輯直接下顧問建議結論。
