# document-template-pattern-extractor｜公司範例模板規則萃取器

> 版本：v0.1.0｜狀態：試行中（尚未完成實測）
> 主要用途：範本規則萃取。品質檢查為延伸用途，第一版不建立完整流程。

## 使用時機

當使用者提出以下任一需求時，啟用此 Skill：

- 希望 AI 員工「讀懂」某份公司範例，作為後續文件產出的格式依據
- 需要建立某類文件的撰寫規則，供日後重複使用
- 需要確認某份範本的「寫法邏輯」，例如顧問語氣、欄位必填規則、章節順序

---

## 標準工作流程

收到萃取需求後，**依序執行，不跳步驟**。

### Step 1：確認萃取對象

詢問或確認以下資訊：
- 要萃取的範本檔案路徑或名稱（通常位於 `03_公司範例模板/`）
- 本次萃取的目的（例如：作為某類文件的撰寫基準、建立通用規則）
- 若有多份範本，是否需要跨範本比對，找出共同規則

### Step 2：讀取範本內容

使用 Read 工具讀取指定範本檔案。

若為 .docx、.xlsx 或 .pptx，應使用目前環境可用的讀取工具或既有 Skill；
若無法直接讀取，須回報使用者並請求轉成可讀格式（如純文字或 PDF），
**不得自行猜測或假設範本內容**。

### Step 3：萃取六大面向

逐一分析以下六個面向，不得跳過：

| 面向 | 說明 |
|---|---|
| **章節結構** | 文件包含哪些章節？章節順序為何？哪些章節為必要、哪些為選填？ |
| **欄位設計** | 各章節包含哪些欄位？欄位類型（文字、日期、勾選、表格）為何？ |
| **填寫規範** | 各欄位有無填寫格式要求？例如日期格式、字數限制、必填條件？ |
| **顧問語氣** | 文件使用的語氣風格為何？（如：正式報告體、條列式說明、問答式記錄） |
| **資訊呈現方式** | 資訊如何組織？（如：表格、段落、條列、圖示） |
| **品質標準** | 範本隱含哪些品質期待？（如：需具體可追蹤、禁用模糊詞彙、需附佐證） |

若發現範本本身有矛盾或不一致之處，須標示「待確認」並回報使用者。

### Step 4：輸出撰寫規則文件

將萃取結果整理為結構化的「文件撰寫規則」，使用以下格式：

```
# {文件類型名稱} 撰寫規則
來源範本：{檔案名稱或路徑}
萃取日期：{日期}
狀態：草稿，待使用者確認

## 章節結構規則
## 欄位設計規則
## 填寫規範
## 語氣與風格
## 資訊呈現方式
## 品質標準
## 注意事項
```

**注意事項欄位必須包含：**
- 不得將範本中的客戶案例、金額、組織名稱等具體內容寫入撰寫規則

### Step 5：確認輸出位置

萃取結果預設輸出至 `05_暫存區/` 或任務包草稿區。

只有在使用者確認「可沉澱為共用撰寫規則」後，才可寫入
`顧問Cowork/00_共用規則與知識庫/` 的適當位置。

**不得回寫至 `03_公司範例模板/`（原始範本區禁止 AI 寫入）。**

## 延伸用途：格式與品質檢查（待擴充）

當以下條件均成立時，再將品質檢查擴充為正式 Skill 流程：

1. 已完成至少 1 份範例文件格式萃取
2. 小果已依萃取規則產出過文件草稿
3. 使用者需要比對草稿是否符合公司範例格式
4. 發現文件常有章節缺漏、語氣不一致、欄位漏填問題
5. 同一類文件檢查需求出現 2 次以上

屆時可擴充為：
```
document-template-pattern-extractor
├─ 萃取模式（現有）
└─ 品質檢查模式（待建立）
```

第一版不建立品質檢查模式。如有需要，以人工對照萃取規則為主。

## 輸出規則

- 萃取結果必須標示「來源範本」與「萃取日期」
- 萃取結果屬草稿，須經使用者確認後才能移至正式知識庫
- 若資料不足或無法判斷，標示「待確認」，不得自行推測
