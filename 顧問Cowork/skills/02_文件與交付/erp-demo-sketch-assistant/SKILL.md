# erp-demo-sketch-assistant｜ERP 顧問 Demo 前準備助理

> 版本：v0.1.0｜狀態：試行中
> 主要用途：ERP 顧問在對客戶進行 Odoo 模組展示前，設計展示流程順序、說明腳本草稿與前置條件確認清單。

---

## 一、Skill 定位

### 本 Skill 做什麼

協助 ERP 顧問在 Demo 前完成以下三件事：

1. **展示流程設計**：規劃展示步驟順序、對應模組畫面與展示重點
2. **說明腳本草稿**：為每個展示步驟提供面向客戶的口頭說明摘要
3. **前置條件確認清單**：列出 Demo 前須確認完成的準備項目（供顧問自行核對）

### 本 Skill 不做什麼

- ❌ 不取代 `odoo-ui-analyzer`（截圖分析工具，本 Skill 的上游輸入）
- ❌ 不取代 `odoo-module-spec-writer`（讀原始碼產規格書，完全不重疊）
- ❌ 不取代外層 `design-sketch`（通用 PPT 架構 / 流程圖草圖）
- ❌ 不產出 .pptx 投影片、mockup 圖片或正式交付文件
- ❌ 不執行資料建置、系統設定或任何 Odoo 操作
- ❌ 不產出正式需求規格書、GAP 分析結論或 UAT 測試腳本

### 架構定位

```
【上游】
odoo-ui-analyzer（截圖分析）→ 可提供畫面結構輸入
erp-interview-researcher（訪談問題）→ 確認客戶現況後進入 Demo
erp-task-planning-assistant（任務規劃）→ 確認 Demo 任務範圍

【本 Skill】
erp-demo-sketch-assistant
↓ 產出：Demo 草圖 + 腳本草稿 + 前置條件清單（顧問內部工作文件）

【下游】
erp-meeting-record-writer（Demo 後會議紀錄）
anthropic-skills:pptx + design-sketch（正式投影片製作）
員工_01（需求確認與 GAP 分析）
```

## 二、解決的 ERP 顧問工作問題

| 問題 | 本 Skill 如何協助 |
|---|---|
| Demo 前不知道展示順序如何安排 | 依模組流程設計有邏輯的展示步驟序列 |
| 口頭說明思路不清，容易跳太快或太慢 | 提供每步驟的腳本摘要，協助顧問預演說明節奏 |
| Demo 前不確定哪些東西要先準備好 | 產出前置條件確認清單，讓顧問自行核對 |
| 客戶看不懂技術術語，顧問難以轉換語氣 | 腳本草稿採業務語言，避免 Odoo 技術術語 |
| 多模組 Demo 時容易遺漏某個展示環節 | 結構化表格確保每個步驟有對應模組與重點 |

## 三、適用情境

- ERP 顧問需要在客戶會議前準備 Odoo 模組展示
- 需規劃單一或多模組的展示流程順序
- 需為每個展示步驟準備口頭說明草稿
- 需在 Demo 前確認環境、資料、帳號等準備狀況
- 基於 `odoo-ui-analyzer` 分析輸出，進一步設計展示腳本

## 四、不適用情境

| 情境 | 應改用 |
|---|---|
| 分析客戶現有 Odoo 畫面截圖 | `odoo-ui-analyzer` |
| 讀 Odoo 模組原始碼產出功能說明書 | `odoo-module-spec-writer` |
| 製作正式 PowerPoint 投影片或簡報檔 | `anthropic-skills:pptx` + 外層 `design-sketch` |
| Demo 後整理會議紀錄 | `erp-meeting-record-writer` |
| 確認 Demo 需求範圍、任務分工 | `erp-task-planning-assistant` |
| 製作客戶驗收腳本或 UAT 測試案例 | 員工_01（需求與規格）|
| 正式需求規格書或 GAP 分析 | 員工_01（專案分析師）|
| 繪製通用流程圖或 PPT 架構草圖 | 外層 `design-sketch` |
| 執行 Demo 環境資料建置 | 由顧問自行操作或另行授權 |

> **本 Skill 只處理 Demo 前準備，不處理 Demo 後事項。** Demo 後的會議紀錄、需求確認結論、系統設定操作，均不在本 Skill 範圍內。

## 七、預設輸出格式

所有輸出均為對話中的草稿，預設不寫入任何檔案。

### 7.1 輸出結構

```
【ERP Demo 草圖｜{企業代號}｜{模組或主題}｜{日期}】
⚠️ 草稿狀態：本輸出為顧問內部工作草稿，不得直接作為客戶正式簡報或交付物

一、展示流程表格
二、關鍵說明詞彙建議
三、前置條件確認清單
四、本次 Demo 目標確認項目
```

### 7.2 格式規則

- 展示流程表格以步驟為主軸，每步驟包含：模組畫面、展示重點、說明腳本摘要、預計時間、前置條件
- 若 `odoo-ui-analyzer` 輸出已提供，可在展示流程表格的「展示重點」或「前置條件」欄中補充具體操作路徑；未提供時，依使用者提供的 Demo 目的與模組範圍填寫，不得補造截圖、畫面文字或系統操作路徑
- 說明腳本摘要採業務語言，**避免使用 Odoo 技術術語**（如 model name：`sale.order`、`purchase.order`；或 server action、automated action、onchange、constraint 等開發層名稱）；**可使用客戶能理解的系統功能名稱**（如：採購訂單、確認按鈕、收貨記錄、結帳流程、廠商主檔）
- 時間估算為參考值，應標示「預計」並說明可依現場調整
- 本 Skill 不補造 Demo 資料內容、截圖或系統畫面文字；若客戶 Demo 資料未知，應在前置條件清單標示「建議確認是否已建立」

### 7.3 展示流程表格

```
| 步驟 | 模組 / 畫面 | 展示重點 | 說明腳本摘要 | 預計時間 | 前置條件 |
|---|---|---|---|---|---|
| 1 | | | | | |
| 2 | | | | | |
...
```

### 7.4 關鍵說明詞彙建議

列出本次 Demo 中建議統一使用的業務術語對照，例如：

```
- 「採購單」而非「purchase.order」
- 「確認採購」而非「Confirm」按鈕操作
- 「廠商」而非「partner」
```

### 7.5 前置條件確認清單

> ⚠️ **本清單為供顧問自行確認的提示清單，不由本 Skill 執行任何資料建置、系統設定或 Odoo 操作。**
> 所有「建議確認是否已建立」項目均由顧問在 Demo 前自行處理或另行安排。

常見確認項目（依實際輸入調整，非固定清單）：

```
□ Demo 環境 / 帳號是否可正常登入
□ 目標模組是否已安裝並可操作（若為 Demo 環境）
□ 展示所需廠商 / 客戶 / 品項主檔：建議確認是否已建立
□ 展示所需採購單 / 銷售單草稿：建議確認是否已建立
□ 報表 / 查詢畫面所需的歷史資料：建議確認是否已備妥
□ 若為正式環境 Demo：⚠️ 避免在正式資料中建立測試主檔或測試訂單，確認展示路徑不影響正式資料
□ 投影設備 / 網路連線是否正常
□ 顧問展示帳號權限是否包含本次展示功能
```

**若「展示環境」為正式環境，需在清單頂端加入以下警示：**
```
⚠️ 正式環境 Demo 注意：請確認展示操作不會建立或污染正式生產資料。
   建議僅展示查詢與表單瀏覽，避免在正式環境執行「確認」、「驗證」、「過帳」等不可逆操作。
```

### 7.6 本次 Demo 目標確認項目

> **本欄位列出本次 Demo 預計與客戶確認的問題面向，供顧問在展示過程中適時提問。**
> 本欄位不作為正式需求結論或知識庫更新依據；Demo 後的需求整理應由員工_01 另行執行。

```
- 此流程是否符合貴公司的採購作業方式？
- 是否有不適用的欄位或缺少的欄位？
- [依模組與 Demo 目的補充]
```

## 十、風險點

| 代號 | 風險 | 緩解方式 |
|---|---|---|
| R-001 | Demo 草圖被誤當正式需求確認結論或規格文件 | 輸出頂端固定標示「草稿狀態」；7.6 明確說明「本次 Demo 目標確認項目」不作為正式需求結論；寫檔限制禁止存入正式交付區 |
| R-002 | 跨企業 Demo 腳本污染（將 A 企業 Demo 腳本誤用於 B 企業） | 每次啟動前確認企業代號；企業代號未確認時草稿標示暫定；禁止跨企業引用設定或腳本 |
| R-003 | 補造 Demo 資料內容（憑空產出客戶未提供的測試資料或畫面文字） | 格式規則明確說明「不補造 Demo 資料內容」；若資料未知一律標示「待確認」或「建議確認是否已建立」 |
| R-004 | 與外層 `design-sketch` 觸發混淆 | 觸發詞規則明確排除「Demo 畫面建議」；「Demo 腳本草稿：」不帶 ERP 信號不觸發；8.快速對照表提供觸發詞歸屬判斷 |
| R-005 | Demo 草稿被誤當 UAT 驗收測試腳本 | 不適用情境明確列出「客戶驗收腳本 → 員工_01」；草稿標示不作為 UAT 依據 |
| R-006 | 說明腳本出現 Odoo 技術術語，客戶無法理解 | 格式規則明確列出禁用術語（model name、server action 等）與可用名稱（採購訂單、確認按鈕等）；業務語言規則精確定義邊界 |
| R-007 | 使用者要求產出 .pptx 或 mockup 圖片 | 寫檔限制明確禁止；回應時提示改用 `anthropic-skills:pptx` + `design-sketch` |
| R-008 | 前置條件確認清單被誤解為 AI 應執行的資料建置任務清單 | 7.5 清單頂端固定標示「本清單為供顧問自行確認的提示清單，不由本 Skill 執行任何資料建置、系統設定或 Odoo 操作」；所有待準備項目均使用「建議確認是否已建立」而非「需提前建立」；不適用情境明確列出「執行 Demo 環境資料建置 → 由顧問自行操作或另行授權」 |

## 十一、回復方式 / 停用方式

若安裝後觸發條件出現錯誤，或與既有 Skill 觸發混淆：

1. 停止使用本 Skill
2. 向使用者回報問題描述（混淆的 Skill 名稱與觸發情境）
3. 提出停用 / 封存計畫，等待使用者授權
4. 授權後將 `skills/02_文件與交付/erp-demo-sketch-assistant/` 移至 `skills/99_封存/`
5. 若 `02_文件與交付/` 內已無其他 Skill，一併評估是否移除空資料夾（需另行授權）
6. 同步更新 `skills/README.md` 索引與相關壓測紀錄狀態
