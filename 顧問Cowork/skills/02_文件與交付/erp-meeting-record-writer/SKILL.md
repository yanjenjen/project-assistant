# erp-meeting-record-writer｜ERP 顧問輔導備忘錄產出器

> 版本：v0.1.0｜狀態：試行中（尚未完成多場景壓測）
> 格式依據：WR-002 v0.2 已壓測試行版（`00_共用規則與知識庫/02_共用模板/會議紀錄撰寫規則_v0.2_已壓測試行版.md`）
> 主要用途：ERP 輔導會議原始素材 → 正式備忘錄草稿。不涉及 Odoo 畫面分析或規格書產出。

---

## 一、Skill 定位

將 ERP 顧問輔導會議的原始素材，依 WR-002 v0.2 四段式格式（A 標頭 + B 討論 + C 決議事項 + D 待辦追蹤）整理為可交顧問人工確認的備忘錄草稿。

**在工作區 Skill 架構中的位置：**

```
document-template-pattern-extractor   ← 上游：從公司範例萃取撰寫規則（WR-002 的來源）
        ↓
erp-meeting-record-writer（本 Skill）  ← 中游：依規則將素材產出草稿
        ↓
consultant-output-evaluator            ← 下游：評核草稿品質
```

**本 Skill 只產出草稿，不做最終判斷。** 流程判斷、決議是否成立、待辦責任方是否正確，均需執行顧問人工確認。

---

## 二、適用情境

- ERP 顧問輔導會議後，需依 WR-002 格式整理正式備忘錄草稿
- 素材來源為 VTT 字幕、錄音逐字稿、口述摘要或手寫筆記
- 需快速產出 A+B+C+D 四段式結構，含 TO-001 表格與 D-001 決議編號
- 多議題會議需整理成層級條列（1 / 1.1 / 1.1.1）
- 素材資訊不完整時，需就地標示 `[待確認]` 並於文末彙整待確認清單

**含客戶方出席的會議，或輸出需遞交客戶確認的備忘錄，請使用本 Skill。**

---

## 三、不適用情境

- **一般非 ERP 會議**（日常工作討論、課程摘要、主管交辦）→ 使用外層 `meeting-summary`
- **顧問方純內部討論**（無客戶出席、輸出無需交付客戶）→ 使用外層 `meeting-summary`
- **正式規格書產出** → 非本 Skill 職責
- **Odoo 畫面分析** → 使用 `odoo-ui-analyzer`
- **多份不同場次會議合併整理** → v0.1 不支援，請使用者拆分（見第五節限制）
- **直接交付客戶的正式文件** → 本 Skill 輸出均為草稿，需人工確認後方可交付

---

## 四、觸發詞

**主要觸發詞（推薦使用，意圖最明確）：**
- `erp-memo：`
- `整理這次 ERP 輔導備忘錄`
- `幫我寫 ERP 輔導備忘錄`

**輔助觸發詞（語境明確為 ERP 輔導場景時啟用）：**
- `依 WR-002 整理`
- `把這份素材整理成 ERP 輔導備忘錄`
- `整理這次輔導紀錄`

> 設計原則：觸發詞必須帶有「ERP」、「輔導」或「備忘錄」語意，與外層 `meeting-summary` 的觸發詞（`會議記錄：`、`幫我做一份會議紀錄`）明確區隔。不確定時，建議使用 `erp-memo：` 前綴確保正確觸發。

## 五、輸入資料要求

### 5.1 輸入項目表

| 輸入項目 | 必填 / 選填 | 說明 |
|---------|------------|------|
| **企業代號或客戶名稱** | **必填** | 用於文件識別與檔名首碼；未提供時必須先詢問使用者確認；確認後才進入產出流程。若使用者明確表示暫不提供，才可標示 `[待確認：企業代號]` 並繼續，不得靜默省略。 |
| **會議素材** | **必填** | VTT 字幕、逐字稿、口述摘要、手寫筆記均可；素材不完整時就地標 `[待確認]` |
| 客戶正式名稱 | 建議提供 | 不確認則填 `[待確認：客戶全名]`，不得自行猜測 |
| 會議日期與時間 | 建議提供 | 格式 `YYYY/MM/DD HH:MM～HH:MM`；不確認填 `[待確認]` |
| 參與人員 | 建議提供 | 採三層格式（公司別 / 部門 / 姓名）；層缺少時標 `[待確認]` |
| 執行顧問 | 建議提供 | 不確認填 `[待確認]` |
| 計費類型 | 選填 | 說明是否為計費型輔導；影響時數欄填法；不提供填 `[待確認]` |

### 5.2 v0.1 素材限制

- **一次只處理單次會議素材。** 若使用者提供多場會議素材，需請使用者拆分為單次，或明確確認是否合併（合併需使用者授權，Skill 不自行決定）。
- 素材語意不清、語音辨識錯誤、或邏輯中斷時，一律標示 `[待確認]`，不得補造內容。
- 本 Skill 不做系統設定判斷、流程設計建議，這類內容應標 `[待確認]` 或列入待辦。

### 5.3 禁止補造

以下資訊若素材中未出現，一律標 `[待確認]`，不得自行填入：

- 人名、職稱、部門
- 決議結論、承諾事項
- 技術參數、系統設定
- 期限日期、責任歸屬

## 六、預設輸出格式

輸出格式完整依 WR-002 v0.2 四段式版型，並附文末待確認清單（Skill 輔助功能）。

```
【ERP 輔導備忘錄】草稿｜需人工確認後方可交付

會議主題：
客戶名稱：
會議日期：YYYY/MM/DD HH:MM～HH:MM
參與人員：
  - 公司別 / 部門 / 姓名（缺少資訊層填 [待確認]）
執行顧問：
輔導時數：（計費型：X.XH；非計費型：不適用；不確定：[待確認]）
交通時數：（計費型：X.XH 或 0H；非計費型：不適用；不確定：[待確認]）

1. （第一議題）
  1.1. （說明 / 結論）
  1.2. ...
2. （第二議題）
  ...

【決議事項】

D-001：（決議內容，記述式，無主觀語氣）
（若無決議）本次會議未形成明確決議，相關事項列入【待辦追蹤】或標示為 [待確認]。

【會後補充-主題說明】（條件選填，非決議非待辦才使用）
（說明或補充內容）

【待辦追蹤】

| 編號   | 責任方     | 待辦事項         | 期限       | 狀態   |
|--------|-----------|----------------|------------|--------|
| TO-001 | (責任方)  | （可執行的動詞句） | [待確認]   | 待處理 |

---
【待確認清單】（Skill 輔助功能，非 WR-002 格式規定）

以下項目因素材不足或語意不明，尚待確認，請執行顧問逐一補齊：
- [待確認-01]：
- [待確認-02]：
```

> **重要說明：**
> - `[待確認]` 採雙軌標示：文中缺漏處就地標示，文末另以清單彙整，兩者不得省略其一。
> - 文末「待確認清單」為 Skill 輔助功能，非 WR-002 格式規定的一部分，顧問確認完成後可刪除。
> - 【決議事項】區塊不得省略，有無決議均需填寫。

## 七、決議判斷規則

依素材語意三段式判斷：

| 素材語意 | Skill 行為 |
|---------|-----------|
| **明確決議語意**（如：確定採用 X、決議 Y 模式先上線、大家同意 Z 方案） | 整理為 D-001，以記述式呈現，不加主觀語氣 |
| **模糊或隱性共識**（如：感覺大家都贊成、應該是這個方向、好像可以這樣） | 標示 `D-001：[待確認：是否已形成正式決議]`，不得逕行判定 |
| **完全無決議語意** | 填 WR-002 固定說明句：「本次會議未形成明確決議，相關事項列入【待辦追蹤】或標示為 [待確認]。」 |

> 不得將語意模糊內容判定為正式決議。決議內容應以記述式呈現，不加主觀推測語氣。

## 九、寫檔限制

- **預設只在對話中輸出**，不自動寫入任何檔案
- 不自動寫入 worklog.md、README.md、正式交付成果區（90_）或跨員工協作區任務包
- 若使用者授權寫入，需另行確認：
  - **企業代號**（必須確認，避免跨企業資料污染）
  - **寫入路徑**（任務包草稿區或 05_暫存區，不得直接寫入 90_正式交付成果）
  - **草稿或正式版狀態**（v0.1 輸出一律為草稿）
- **檔名命名規則：** 若授權寫入，檔名必須以企業代號或專案代號為首碼
  - 格式：`{企業代號}_輔導備忘錄_{YYYYMMDD}_Draft.md`
  - 範例：`建大_輔導備忘錄_20260611_Draft.md`
  - 不得使用不含企業識別的通用檔名（如 `備忘錄_Draft.md`）
- 輸出文件標頭必須標示「草稿｜需人工確認後方可交付」

## 十、風險點

| 編號 | 等級 | 風險描述 | 對策 |
|------|------|---------|------|
| R-001 | 🔴 高 | **誤當正式文件交付**：草稿未經人工確認就遞交客戶 | 標頭固定加「草稿｜需人工確認」聲明；文末待確認清單讓未確認狀態一眼可見 |
| R-002 | 🔴 高 | **跨企業資料污染**：素材企業來源與當前任務包企業代號不一致，產生錯誤輸出 | 啟動時確認素材企業來源；若企業來源不明或與任務包不一致，暫停並提示使用者確認後再執行 |
| R-003 | 🟠 中 | **素材不足時自行補造**：VTT 字幕語意不清，模型補填內容 | SKILL.md 明定禁止補造；資訊不足一律標 `[待確認]`；文末清單集中列出讓顧問核對 |
| R-004 | 🟠 中 | **決議判斷過度主觀**：模糊共識被認定為正式決議 | 採三段式判斷規則（第七節）；模糊語意一律標 `[待確認]`，不逕行判定 |
| R-005 | 🟡 低 | **與 meeting-summary 觸發混淆**：一般摘要需求意外啟動本 Skill | 觸發詞設計要求「ERP / 輔導 / 備忘錄」語意；建議使用 `erp-memo：` 前綴 |
| R-006 | 🟡 低 | **WR-002 格式限制繼承**：WR-002 v0.2 僅完成一次壓測，本 Skill 繼承同樣格式不確定性 | SKILL.md 標注「格式依據 WR-002 v0.2，本 Skill 輸出品質受 WR-002 版本限制」；WR-002 升版時本 Skill 需同步評估是否調整 |

## 十一、回復方式 / 停用方式

若觸發條件出現錯誤、輸出格式與 WR-002 規則衝突、或被誤用為正式交付文件：

1. 停止使用本 Skill
2. 向使用者回報問題描述（觸發情境、誤用方式、與 WR-002 的衝突點）
3. 提出停用 / 封存計畫，等待使用者授權
4. 授權後將 `skills/02_文件與交付/erp-meeting-record-writer/` 移至 `skills/99_封存/`
5. 若 `02_文件與交付/` 內無其他 Skill，一併評估是否移除空資料夾（需另行授權）
6. 同步更新 `skills/README.md` 索引狀態
