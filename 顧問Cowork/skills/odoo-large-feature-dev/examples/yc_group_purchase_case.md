# 範例：總部代辦品採購（yc_group_purchase）

## 背景
四季藝術幼兒園集團，5 校 + 總部，集團團購流程。

## 階段 1：需求拆解

5 大業務階段：
- A：各校提報需求
- B：總部統一採購
- C：商品入庫到各校專屬倉
- D：各校領貨給家長
- E：月底總部彙整對帳

## 階段 2：5 大關鍵收斂

| # | 議題 | 決定 |
|---|---|---|
| 1 | 家長 partner | 1 個共用 |
| 2 | 月底 SO | 每校 1 張 |
| 3 | 校區待收款 | 內部 invoice |
| 4 | 校區倉庫 | location 區分（不用 warehouse） |
| 5 | 品項主檔 | 統一 |

## 階段 3：規格回放發現

Excel 上額外挖到的：
- 需求量是公式估算（在校生×1.5 + 新生問卷 + 缺額）
- 教師品項用 老師×比例 公式（不同於學生）
- 寶貝袋用 (新生 - 在校生×5%) + 50 公式
- 跨校調貨：彙整時即會發生（市政>北屯6 仁美>黎明1）
- 統採 100 件批發價須加購 28 件
- 總部多備量
- 5/13 追加版本
- 教師/學生品項分流（不同 sheet）

## 階段 4：細節決定

- 公式：4 預設類型 + 自由表達式
- 歷史銷售：反查 + CSV 匯入
- 跨校調撥：獨立 model + 彙整單快捷入口
- 批發補量：人工調整欄位（不自動分攤）
- 追加：新單據（is_supplement）
- 教師/學生分流：product 標記 + 單據 user_type

## 階段 5：技術選型

- 跨校調撥 done：B 方案（自動建 picking + auto validate）
- 代辦品申請單也走 B
- 月底結算：模組自建兩張 invoice + 自建出貨 picking
- SO 不 confirm（避免重複認列）
- partner company_id 共用

## 階段 6：執行計畫
- 16 model（13 自定 + 5 繼承 + 1 wizard）
- 12 view
- 模組命名 yc_group_purchase
- 一次做完（不分階段交付）

## 階段 7：開發
- 2,262 行 Python
- 1,130 行 XML
- 11 個 task 依序完成

## 階段 8：驗證
- 49 單元測試全通過
- 11 E2E 驗證全通過
- 修了 1 個 depends 鏈漏

## 階段 9：交付
- 模組程式（兩處同步）
- 測試流程文件 .docx
- 3 個匯入 Excel/CSV
- 兩支測試腳本

## 完成時間估算
從需求 → 完整交付 ≈ 1 個工作天（含對話來回）
